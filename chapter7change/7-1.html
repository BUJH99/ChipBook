# TIME_CONSUMPTION_ANALYSIS

1.1 Function vs Task (0-Time의 본질)

검증 환경에서 로직을 작성할 때 가장 치명적인 버그는 타이밍 의도가 어긋날 때 발생합니다. function은 시뮬레이터의 시간을 완벽히 동결(Freeze)시킨 상태에서 모든 연산을 즉시 끝내야 하는 '소프트웨어적' 계산 영역입니다. 반면 task는 클럭을 기다리고 신호를 인가하며 시뮬레이션 시간을 앞으로 진행시키는 '하드웨어적' 상호작용 영역입니다.

WAVEFORM VIEWER
SYS_TIME: 0 ns

Function의 제약 (0-Time)

build_phase 등은 모두 function입니다. 내부에서 #10, @(posedge) 등을 사용하면 컴파일 에러가 발생합니다. 시뮬레이터는 이 함수를 만나는 순간 전체 시스템 시계를 정지시키고 연산을 수행합니다. 

function void calc_parity(input int data);
  // #10;  <-- FATAL ERROR: Delay in function
  parity_bit = ^data; // 즉시 연산 완료
endfunction

Task의 권한 (Time Consuming)

run_phase와 같이 실제 핀(Pin)을 토글하거나 응답을 대기해야 하는 영역은 task입니다. 시뮬레이션 시간이 정상적으로 흐르며, 내부에 다른 task를 호출할 수 있습니다. 

task drive_packet(input Packet pkt);
  @(posedge vif.clk); // 시간 흐름 (Wait)
  vif.data <= pkt.data;
  #10ns;              // 시간 흐름 (Delay)
endtask
# INHERITANCE_AND_POLYMORPHISM

1.2 & 1.3 가상 메서드(Virtual)와 Super

UVM 벤치를 구성하는 핵심은 "프레임워크가 짜놓은 로직(Base) 위에 내 로직(Extended)을 덧칠하는 것"입니다. virtual 키워드는 실제 자식의 메서드를 찾아가게 해주는 나침반 역할을 하며, super는 자신의 로직 실행 도중 잠시 부모의 영역으로 올라가 원본 로직을 실행하고 내려오게 합니다.

CALL_STACK_MONITOR
"Packet..."); extends Extended: class ErrorPacket bit error_flag; function display(); super.display(); $display("Error...");

Virtual의 의미 (다형성)

베이스 클래스의 핸들(포인터)로 확장 클래스 객체를 가리킬 때, virtual이 없으면 컴파일러는 무조건 베이스 클래스의 메서드를 호출합니다. virtual 선언을 통해 "실제 메모리에 생성된 객체 타입(Run-time Type)"을 추적하여 알맞은 자식 메서드를 실행하게 만듭니다. UVM의 Phase 메서드가 모두 virtual인 이유입니다.

Super의 호출 규칙

super.build_phase(phase) 처럼 호출하면, 내가 작성한 자식 클래스의 코드가 잠시 멈추고 제어권이 부모 클래스로 넘어갑니다. 부모가 UVM 코어 로직을 안전하게 세팅하고 나면, 다시 제어권이 내려와 나의 커스텀 로직을 마저 실행합니다. 원본 보존의 핵심 메커니즘입니다.

# MEMORY_ALLOCATION_LIFECYCLE

1.4 Class(동적 할당) vs Module(정적 할당)

반도체 검증에서 수백만 개의 트랜잭션 패킷을 생성해야 할 때, 하드웨어 module로는 불가능합니다. Module은 컴파일 타임에 고정(Static)되기 때문입니다. 반면 class는 런타임에 new()를 통해 힙(Heap)에 끊임없이 할당되고, 수명이 다하면 Garbage Collector (GC)에 의해 자동 회수됩니다.

# BRIDGING_STATIC_AND_DYNAMIC

1.5 Virtual Interface와 uvm_config_db

동적인 세계(UVM Class)에서는 정적인 세계(Hardware Module/Interface)를 직접 생성하거나 포함할 수 없습니다. 이 두 세계를 연결하는 유일한 다리가 바로 Virtual Interface (인터페이스의 메모리 포인터)이며, 이 포인터를 안전하게 배달해주는 우체국 역할이 uvm_config_db입니다.

Top Module (Static) db::set() 

module top;
  // 1. 물리 인터페이스 실체 생성
  my_if vif_inst(clk);
  
  DUT dut(.in_data(vif_inst.data));

  initial begin
    // 2. DB에 포인터 주소를 등록
    uvm_config_db#(virtual my_if)::set(
      null, "*", "vif", vif_inst
    );
    run_test();
  end
endmodule

UVM Driver (Dynamic) db::get() 

class my_driver extends uvm_driver;
  virtual my_if vif; // 빈 포인터 선언

  function void build_phase(...);
    // 3. DB에서 찾아서 포인터에 연결!
    uvm_config_db#(virtual my_if)::get(
      this, "", "vif", vif
    );
  endfunction
  
  task run_phase(...);
    vif.data <= 8'hFF; // 하드웨어 구동
  endtask
endclass

1.6 Constrained Random (제약 기반 난수 생성)

검증의 커버리지를 극대화하기 위해 단순히 무작위 값을 넣는 것이 아니라, 프로토콜에 위배되지 않는 유효한 값(Constraint) 내에서 난수를 생성합니다. rand는 독립적 확률로 값을 뽑아 중복이 허용되며, randc(Cyclic)는 모든 경우의 수를 한 번씩 다 뽑을 때까지 중복을 방지합니다.

class Config;

rand bit [1:0] a;
randc bit [1:0] b;

고급 Constraint Block 작성 기법

실무에서는 단순 범위 지정(inside)을 넘어, 조건부 제약(->)이나 가중치 분배(dist)를 결합하여 정교한 프로토콜 스펙을 Testbench에 모델링합니다. 

class MemoryTrans extends uvm_sequence_item;
  rand bit [31:0] addr;
  rand bit [31:0] data;
  rand bit        is_write;

  constraint c_addr_align {
    // Word-aligned 강제 및 특정 메모리 영역 지정
    addr[1:0] == 2'b00;
    addr inside {[32'h1000 : 32'h1FFF]};
  }
  
  constraint c_read_protocol {
    // Implication: Read 동작일 경우 data는 0으로 고정
    (is_write == 0) -> (data == 0);
  }

  constraint c_weight {
    // Write 커맨드가 발생할 확률을 70%로 부여
    is_write dist { 1 := 70, 0 := 30 };
  }
endclass
