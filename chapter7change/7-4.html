공장(Factory)과 제품(Object)의 분리

UVM을 완벽히 다루는 핵심은 환경을 구성하는 '설비(Component)'와 그 안에서 처리되는 '데이터 패킷(Object)'을 명확히 분리하여 사고하는 것입니다.

🏛️

uvm_component

정적 아키텍처 (Quasi-Static)

  • 계층 구조 (Hierarchy): 부모-자식 관계를 가지며 트리를 형성합니다. (예: Env -> Agent -> Driver)
  • 생명 주기 (Phase): 시뮬레이션 시작에 한 번 생성(build)되어 끝날 때까지 메모리에 상주합니다.
  • 연결 (TLM Port): 컴포넌트 간에는 Port/Export를 통해 전용 통신망을 연결합니다.
✉️

uvm_object

동적 데이터 (Dynamic)

  • 트랜잭션 (Transaction): Sequence Item과 같이 버스를 타고 다니는 실질적인 데이터 패킷입니다.
  • 생명 주기 (Transient): 시뮬레이션 중 필요할 때마다 동적 생성(new)되고 소멸(GC)됩니다.
  • 계층 없음: 물리적 구조가 아니므로 부모(parent) 파라미터를 갖지 않으며 Phase 메커니즘도 없습니다.

정적 구조 설계: Agent 재사용성 극대화

Agent는 인터페이스 통신의 기본 단위입니다. UVM은 코드 수정 없이 Config DB 설정만으로 Agent 내부 구조를 Active(직접 구동) 또는 Passive(수동 모니터링) 모드로 변형시켜 재사용성을 100% 확보합니다.

my_agent.sv
class my_agent extends uvm_agent; `uvm_component_utils(my_agent) uvm_active_passive_enum is_active; my_sequencer sqr; my_driver drv; my_monitor mon; // 항상 생성됨 virtual function void build_phase(uvm_phase phase); super.build_phase(phase); // Monitor: Snoop 용도로 항상 Factory 생성 mon = my_monitor::type_id::create("mon", this); // Config에 따라 분기 처리 if(get_is_active() == UVM_ACTIVE) begin sqr = my_sequencer::type_id::create("sqr", this); drv = my_driver::type_id::create("drv", this); end endfunction endclass

동적 데이터 제어: Core Methods 실습

트랜잭션(uvm_object)은 내장 제어 메서드를 가집니다. 필드 매크로(`uvm_field_*)는 코딩이 편하지만 시뮬레이션 속도 저하가 심각하므로, 실무(특히 Scoreboard 로직)에서는 do_copy, do_compare를 직접 오버라이딩하여 성능을 극대화합니다.
