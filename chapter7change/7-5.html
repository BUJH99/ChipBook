Sequence 메커니즘

객체(Object)인 트랜잭션이 컴포넌트(Component) 계층인 Driver까지 전달되어 실제 핀 신호로 변환되는 REQ/RSP 양방향 핸드셰이크 프로토콜의 정밀한 타이밍을 분석합니다.

Sequence ↔ Sequencer ↔ Driver 양방향 흐름

1
start_item

Arbiter에 접근 권한을 요청합니다. 여러 Sequence 경합 시 우선순위에 따라 블로킹됩니다.

2
finish_item

Late Randomize를 수행하고, REQ를 포트를 통해 Driver로 보냅니다.

3
get_next_item

Driver는 무한 루프 속에서 대기하다가 포트로부터 REQ 트랜잭션을 당겨(Pull)옵니다.

4
item_done(rsp)

실제 핀 토글링 완료 후 RSP를 반환하며 호출하면, 블로킹이 해제됩니다.

실무 팁: Late Randomization

start_item과 finish_item 사이에서 randomize()를 호출하는 이유가 있습니다. 대기 시간(Arbitration) 동안 DUT의 상태가 변할 수 있으므로, 인가되기 직전 최신 상태를 반영해 랜덤값을 풀기 위함입니다. 

class my_sequence extends uvm_sequence #(my_tx, my_tx);
  virtual task body();
    req = my_tx::type_id::create("req");
    start_item(req);          // 1. 실행 권한 요청 (블로킹)
    assert(req.randomize());  // 2. Late Randomization
    finish_item(req);         // 3. 전송 및 item_done 대기
    get_response(rsp);        // (Optional) 응답 수신
  endtask
endclass

Virtual Sequence (시스템 시나리오 조율)

복합 SoC 환경에서 하위 물리적 Agent들을 조율하는 지휘자 역할을 합니다.

MACRO `uvm_declare_p_sequencer

모든 Sequence는 내부적으로 자신이 구동되는 Sequencer를 가리키는 m_sequencer 포인터를 가지지만 최상위 베이스 클래스 타입이므로 하위 속성에 접근할 수 없습니다. 이 매크로로 컴파일 타임에 캐스팅된 p_sequencer 핸들을 생성합니다. 

class my_vseq extends uvm_sequence;
  `uvm_declare_p_sequencer(my_vsequencer)
  
  virtual task body();
    ahb_seq_t ahb_seq = ahb_seq_t::type_id::create("ahb");
    apb_seq_t apb_seq = apb_seq_t::type_id::create("apb");

    fork
      ahb_seq.start(p_sequencer.ahb_sqr);
      apb_seq.start(p_sequencer.apb_sqr);
    join
  endtask
endclass

TLM (Transaction Level Modeling)

데이터 객체(Transaction) 통째로 컴포넌트 간에 던져주는 통신 방식으로 시뮬레이션 속도가 향상됩니다.

  • Port : 메서드 호출 주체 (Initiator)
  • Export : 계층 간 연결을 위한 파이프
  • Imp : 실제 메서드 구현 목적지 (Target)

Analysis Port (1:N)

Monitor가 수집한 패킷을 Scoreboard나 Coverage 여러 곳에 동시 브로드캐스팅합니다.

특징: 0-time 실행, 수신자 0개라도 무시(에러없음), 객체 참조(Handle) 전달
