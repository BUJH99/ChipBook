AXI Interconnect가 중앙 허브 역할을 수행합니다. 모든 주소와 데이터는 인터커넥트를 통해 직교 라우팅됩니다.

Software View
#define UART_BASE 0x40001000
volatile uint32_t* reg = UART_BASE;
*reg = 'A'; // Through AXI Fabric

소프트웨어는 I/O를 단순 메모리 접근으로 인식합니다. 인터커넥트가 물리적 주소를 찾아 신호를 라우팅합니다.

AXI + MMIO Synergy
Unified Fabric
모든 컴포넌트가 동일한 AXI 인터페이스를 사용합니다.
Address Decoding
인터커넥트가 주소에 따라 경로를 자동 분배합니다.
AXI Pipelining
I/O 접근 시에도 AXI의 throughput 이점을 그대로 활용합니다.
명령어 세트 효율성

MMIO는 CPU에서 I/O 전용 명령 로직을 제거하게 해 줍니다. LDR/STR만으로 모든 제어가 가능해져 파이프라인 구조가 더 단순해집니다.

접근 제어와 보안

AXI의 AxPROT와 MMIO를 결합하면, 특정 메모리 영역과 I/O 레지스터에 대한 보안/비보안 접근 권한을 fabric 차원에서 통합 제어할 수 있습니다.

비순차 처리 활용

I/O 장치는 메모리보다 응답이 느립니다. AXI의 ID 기반 outstanding transaction은 I/O 응답을 기다리는 동안에도 메모리 접근을 계속 흘려 throughput을 유지합니다.

Address Map 해석 예시

주소 범위는 단순한 메모리 배치가 아니라, 어떤 Slave로 트랜잭션을 라우팅할지 결정하는 하드웨어 규칙입니다.

주소 범위 선택 대상 용도
0x0000_0000 - 0x0FFF_FFFF SRAM / ROM 명령어 및 데이터의 저지연 접근 영역
0x1000_0000 - 0x1FFF_FFFF AHB Peripheral DMA 등 고속 제어 블록 연결 영역
0x4000_0000 - 0x4000_FFFF APB Peripheral UART, GPIO, Timer의 레지스터 접근 영역

실제 SoC에서는 decoder가 이 memory map에 따라 대상 Slave를 선택하므로, 주소 범위는 곧 버스 라우팅 규칙을 의미합니다.

AMBA 버스를 APB / AHB / AXI로 분화하는 이유

하나의 BUS 구조로 모든 요구사항을 만족시키기 어렵기 때문에, AMBA는 대역폭, 복잡도, 전력, 동시성 요구에 따라 프로토콜을 계층적으로 분화했습니다.

APB

레지스터 접근 중심의 저속 주변장치는 구조의 단순성, 구현 용이성, 저전력 특성이 우선되므로 APB가 적합합니다.

AHB

중간 수준의 대역폭과 공유형 시스템 버스가 필요한 구간에서는 burst와 pipeline을 지원하는 AHB 구조가 효율적입니다.

AXI

CPU, DMA, DDR처럼 높은 처리량과 동시 전송이 필요한 구간에서는 독립 채널 기반의 AXI 구조가 필요합니다.
