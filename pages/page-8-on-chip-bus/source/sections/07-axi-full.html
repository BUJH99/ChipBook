Definition & Principle

📚 1. 기본 정의

이전 트랜잭션의 데이터 응답(Data Response)이 완료되기 전에 새로운 주소(Address)를 발행할 수 있는 능력을 의미합니다. '미완결(Outstanding)' 상태의 트랜잭션을 여러 개 유지할 수 있다는 뜻에서 유래되었습니다.

⚙️ 2. 작동 원리

  • Master는 Slave가 Busy한 상태여도 Address Channel을 통해 다음 명령을 미리 전달합니다.
  • Slave 내부의 Request Queue에 명령이 쌓이게 되며, 이는 파이프라이닝(Pipelining) 효과를 발생시킵니다.

💡 3. 설계 이점

Slave의 내부 처리 지연 시간(Memory Latency)을 숨길 수 있습니다. 주소를 미리 전달함으로써 데이터 채널의 유휴 시간(Idle Time)을 제거하여 대역폭 효율을 극대화합니다.

Definition & Principle

🔀 1. 기본 정의

트랜잭션의 요청 순서와 상관없이, 먼저 준비된 데이터부터 먼저 응답할 수 있는 기술입니다. AXI 프로토콜의 핵심 신호인 ID Tag(ARID, RID 등)를 사용하여 각 응답이 어떤 요청에 대한 것인지 식별합니다.

🆔 2. 작동 원리

  • Master는 각 요청에 고유한 ID 번호를 부여하여 보냅니다.
  • Slave는 처리 속도가 다른 여러 내부 자원(SRAM, DRAM 등) 중 먼저 처리가 완료된 순서대로 ID를 붙여 데이터를 반환합니다.

🚀 3. 설계 이점

느린 주변 장치에 대한 접근이 빠른 장치로의 접근을 방해하는 Head-of-Line Blocking 현상을 방지합니다. 시스템의 전체적인 유연성과 실질적인 데이터 처리 속도를 크게 향상시킵니다.

Implementation Design Checklist

ID Tagging Mechanism

Out-of-Order를 가능하게 하는 핵심은 ID Signals입니다.

  • ● ARID: Read Address ID
  • ● AWID: Write Address ID
  • ● RID / BID: Response ID
"마스터는 돌아온 응답의 RID를 확인하여 자신이 보낸 어떤 ARID에 대한 데이터인지 판단합니다."

Ordering Rules

1 Same ID Requirement

동일한 ID를 가진 트랜잭션은 반드시 발행된 순서(In-Order)대로 완료되어야 합니다.

2 Different ID Flexibility

서로 다른 ID 사이에는 순서 보장 의무가 없으므로 Out-of-Order가 적극 허용됩니다.

Hardware Overhead

REORDER BUFFER 마스터는 응답을 정렬하기 위한 로직과 버퍼가 필요합니다.
QUEUE DEPTH 슬레이브는 Outstanding 요청을 보관할 Queue가 필요합니다.

* Outstanding Depth가 늘어날수록 성능은 비례하지만 하드웨어 면적(Area) 또한 비례하여 증가합니다.

Practical Hardware Implementation

Master / Slave Structure

RTL Architecture Guide
Master Side: ID Management & ROB
Mechanism 01 Transaction ID Tracking Table

Master는 발행한 모든 Outstanding 트랜잭션을 추적하기 위한 Internal Scoreboard가 필요합니다.

Tracking Logic
  • 1 ID Pool: 사용 가능한 ID 비트 수 2ID_WIDTH 내에서 고유 ID를 할당하여 발행합니다.
  • 2 Response Matching: RID 가 돌아오면 해당 ID의 상태를 완료로 업데이트하고 리소스를 해제합니다.
  • 3 Reorder Buffer (ROB): CPU와 같이 순서가 중요한 Master라면, OoO로 돌아온 데이터를 원래 순서대로 정렬하기 위한 대규모 Buffer를 유지해야 합니다.
Slave Side: Multi-threaded Back-end
Mechanism 02 Request Queueing & Banking

Slave는 다수의 요청을 동시에 수용하고, 처리 준비가 된 장치부터 응답을 보낼 수 있는 병렬 구조를 가집니다.

Backend Architecture
  • 1 Command FIFO: Address Phase에서 수신한 AR 정보를 저장하여 Outstanding 명령을 유지합니다.
  • 2 Memory Interleaving: 주소 공간을 여러 Bank로 나누어, 한 Bank가 작업 중일 때 다른 Bank가 응답할 수 있도록 설계합니다.
  • 3 Arbiter: 처리된 여러 응답 중 어떤 ID를 먼저 Bus로 실어 보낼지 결정하는 Response Arbiter가 필요합니다.

💥 Contention Failure Scenario

멀티 코어 환경에서 세마포어(Semaphore)가 작동하는 핵심 방어 로직을 보여줍니다.

  • ARLOCK=1 read는 bus를 잠그는 read lock이 아니라, monitor 안에 읽기 예약인 {ID, Addr}를 남기는 동작입니다.
  • 이후 다른 master가 같은 주소에 일반 write를 commit하면 그 읽기 예약이 사라집니다. 그래서 뒤따르는 AWLOCK=1 write는 BRESP=OKAY로 실패를 통보받고, 성공했을 때만 EXOKAY와 함께 메모리를 갱신합니다.

🛡️ TZASC Checking Logic

실제 SoC에서 TrustZone은 TZASC(TrustZone Address Space Controller)라는 전용 하드웨어 블록을 통해 구현됩니다.

  • 트랜잭션이 도착하면 TZASC는 AxADDR를 확인하여 해당 영역의 보안 정책을 확인합니다.
  • 만약 접근하려는 주소가 Secure Only 영역인데, 패킷의 AxPROT[1] == 1 (Non-Secure)라면, 접근 차단 및 Master에게 DECERR 에러 응답을 반환합니다.

🗂️ Bufferable Memory Attribute

중간 버퍼 또는 cache hierarchy가 write를 접수하고 나중에 하위 메모리로 밀어 넣는 정책을 시각화했습니다.

  • 0000 (Device / non-bufferable): peripheral 성격의 접근은 부작용 순서가 중요하므로 하위 target의 완료 시점까지 응답을 늦추는 정책이 흔합니다.
  • 0011은 bufferable + modifiable normal access 예시입니다. 이것만으로 write-back을 명령하는 것은 아니며, interconnect/cache 정책이 허용할 때 중간 buffer 수락 후 Early Response(BVALID)를 반환할 수 있습니다.

⚡ Arbiter Weighting Logic

복잡한 Network-on-Chip(NoC) 환경에서 다수의 마스터가 경합할 때, 트랜잭션의 우선순위(Priority)를 결정합니다.

  • 0x0(최하위)부터 0xF(최상위)까지의 값을 가집니다.
  • 디스플레이나 실시간 오디오와 같이 지연 시간에 민감한 트랜잭션(QoS:F)이 도착하면, NoC 내부 arbiter가 다음 선택 지점에서 우선권을 줄 수 있습니다. 이미 수락된 beat를 반드시 중단한다는 뜻은 아니며, 실제 보장 수준은 fabric 정책에 달려 있습니다.

Semaphore Validation Logic

Exclusive Monitor는 다중 마스터의 경합 상태(Race Condition)를 해결하기 위해 하드웨어 레벨에서 상태 머신을 유지합니다.

$$Write\ Exclusive\ Resp = \begin{cases} EXOKAY, & \text{if ReservationValid } \land ID\_Match \land Addr\_Match \\ OKAY, & \text{otherwise (Exclusive Fail)} \end{cases}$$

다른 write가 같은 주소를 먼저 갱신하면 reservation은 삭제되며, exclusive pair는 동일한 주소와 address-channel 속성을 유지해야 합니다.

AxCACHE[3:0] Bit Allocation

BitMeaningProtocol Note
[0]Bufferable중간 buffer가 write response timing을 최적화할 수 있습니다.
[1]Modifiableinterconnect가 transaction을 merge/split/transform할 수 있음을 뜻합니다.
[2]Read Allocatecache miss 때 read allocation 정책 힌트로 쓰입니다.
[3]Write Allocatecache miss 때 write allocation 정책 힌트로 쓰입니다.

AXI4에서는 allocate bit가 set될 때 AxCACHE[1]도 set되어야 합니다. AxCACHE는 coherency 자체를 보장하지 않으므로, 캐시 일관성이 필요한 시스템은 ACE/CHI 또는 coherent interconnect 정책을 함께 봐야 합니다.
