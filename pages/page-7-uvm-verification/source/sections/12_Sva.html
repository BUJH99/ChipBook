Chapter 04. Verification

SVA : Protocol Rule & Timing Behavior

SystemVerilog Assertions(SVA)를 활용하여 설계의 동작, 타이밍, 프로토콜 규칙을 검증하는 방법을 시각적으로 학습합니다. 텍스트를 넘어 파형과 데이터 흐름을 통해 직관적으로 이해해보세요. 기존의 절차적 검증(Procedural Verification)이 잡아내기 힘든 복잡한 타이밍 이슈와 코너 케이스(Corner Case)를 효과적으로 검출할 수 있습니다.

1 SVA 검증 아키텍처 및 역할

설계 의도와 동작을 감시하는 눈

SVA는 설계가 명세서(Specification)에 정의된 프로토콜 규칙과 타이밍 동작을 정확히 준수하는지 런타임에 확인합니다. Interface 내부나 별도의 Assertion Module에 작성되며, bind를 통해 DUT 내부 구조를 건드리지 않고 연결(Plug-in) 구조로 동작합니다.

이를 통해 시뮬레이션 중 발생하는 버그를 발생 즉시(Fail-fast) 포착할 수 있으며, Formal Verification 도구와 결합하여 수학적 증명을 통한 무결성 검증에도 필수적으로 사용됩니다.

Immediate Assertion

RTL 절차적 블록(always, initial) 내에서 일반적인 if문처럼 조합 논리(Combinational logic) 조건을 즉각적으로 검사합니다. 시간에 따른 흐름은 검사할 수 없습니다.

Concurrent Assertion

클럭 엣지(Clocking event)를 기준으로 시간의 흐름(Sequence)에 따른 동작을 검사합니다. 여러 사이클에 걸친 프로토콜 검증에 주로 사용됩니다.

2 SVA 계층 구조 & 지시어

Bottom-Up 기반의 모듈화 & 재사용 (Code Mapping)

SVA는 단순한 조건식에서 출발하여 복잡한 검증 룰로 확장해가는 상향식(Bottom-Up) 계층 구조를 채택하고 있습니다. 이는 마치 레고 블록을 조립하는 것과 같아서, 한 번 잘 만들어둔 sequence와 property는 모듈화되어 여러 검증 환경에서 재사용이 가능합니다.

  • 1.
    Boolean: req == 1과 같이 단일 클럭에서 평가되는 기초 논리식입니다.
  • 2.
    sequence: Boolean 조건들에 ##N 같은 타이밍 연산자를 결합하여 시간의 흐름(패턴)을 만듭니다.
  • 3.
    property: 시퀀스에 클럭 동기화(@posedge clk), 리셋 무시(disable iff) 등의 환경을 입혀 독립적인 규칙으로 포장합니다.
  • 4.
    Directive: 완성된 프로퍼티를 assert, assume, cover 지시어를 통해 시뮬레이터에 실행 명령을 내립니다.
Directives (지시어) & Verification Flow
assume (입력 제약/필터)

검증 환경이나 시뮬레이터가 의도치 않은(유효하지 않은) 자극(Stimulus)을 DUT에 주입하지 못하도록 사전에 필터링하는 제약(Constraint) 역할을 합니다. 특히 Formal Verification에서 탐색 공간을 줄이는 데 필수적입니다.

assert (출력 검증)

DUT의 상태나 출력 결과가 정의된 Property(규칙)를 항상 만족하는지(True) 지속적으로 확인합니다. 위반 사항이 발생하면 즉시 오류(에러)를 보고하여 설계 버그를 잡아냅니다.

cover (커버리지 측정)

성공/실패를 따지는 것이 아니라, 우리가 의도한 특수한 코너 케이스(Corner Case)나 타이밍 흐름이 시뮬레이션 중 한 번이라도 발생했는지(Hit) 횟수를 카운트합니다.

property & sequence
1
sequence (시퀀스)

req ##1 ack와 같이 한 클럭 또는 여러 클럭에 걸친 신호의 연속적인 타이밍 흐름을 정의하는 기본 단위 블록입니다. 블록 단위로 생성되어 다른 곳에서 재사용이 가능합니다.

2
property (프로퍼티)

sequence들을 Implication(|->) 연산자로 결합하여 인과관계를 설정합니다. 또한 어떤 클럭을 사용할지(Clocking Event), 어떤 조건에서 검사를 무시할지(disable iff)를 정의하여 완전한 룰을 구성합니다.

3 Timing & Implication Operators 시각화

주요 연산자 Waveform 동작 분석

##N |-> |=> [*N]
Overlapping |->

조건(Antecedent)이 참이 되는 바로 그 클럭 사이클에서 결과(Consequent) 평가를 시작합니다.

req |-> ack
Non-Overlapping |=>

조건이 참이 된 그 다음 클럭 사이클부터 결과를 평가합니다. (|-> ##1 과 정확히 동일)

req |=> ack
Delay ##N, ##[M:N]

정확히 N 클럭 지연 후 평가합니다. [M:N]을 사용하면 지정된 윈도우 내 언제든(Eventually) 발생하는지 유연하게 검사할 수 있습니다.

req |-> ##[1:5] ack
Repetition [*N]

신호가 연속으로 N 클럭 동안 참임을 나타냅니다. [*]는 0번 이상, [+]는 1번 이상 연속됨을 의미합니다.

req |=> data [*3]

4 Sampled Value Functions

신호의 상태 변화를 포착하는 내장 함수

  • $rose / $fell
    Edge Detection: 이전 클럭 대비 신호가 0에서 1로 상승($rose)하거나, 1에서 0으로 하강($fell)했는지 평가합니다. 인터럽트 발생기나 트리거 조건을 검증할 때 필수적입니다.
  • $stable
    Value Hold: 이전 클럭의 값과 현재 클럭의 값이 동일한지 검사합니다. AXI와 같은 핸드셰이크 프로토콜에서 Ready가 올 때까지 Data가 변경되지 않아야 한다는 규칙(Stable Check)에 주로 쓰입니다.
  • $past(sig, N)
    History Reference: N 클럭 이전의 신호 값을 참조합니다(N 생략 시 1클럭 이전). 복잡한 파이프라인 설계에서 과거의 제어 신호와 현재의 데이터 출력을 비교 대조할 때 강력한 도구입니다.
  • $isunknown
    X/Z Propagation Check: 데이터 버스나 제어 신호에 의도치 않은 X(Unknown) 또는 Z(High-Impedance) 값이 섞여 있는지 검출합니다. 컨트롤 경로의 치명적인 오작동을 사전에 방지합니다.

5 실전 필수 SVA 패턴

1. Ready/Valid Protocol

property p_rv;
  valid && !ready |=> valid;
endproperty

AXI / APB 필수 규칙 Valid가 Assert 되었으나 수신 측에서 Ready를 주지 않았다면(거절/대기 상태), 송신 측은 임의로 Valid를 내리거나 Drop해서는 안 되며 다음 클럭에도 반드시 유지해야 합니다.

2. Data Stable Check

property p_stable;
  valid && !ready |=> $stable(data);
endproperty

데이터 무결성 보장 Valid가 활성화된 상태에서 아직 전송이 완료(Ready)되지 않았다면, 버스에 실린 Payload(Data, Address 등) 값은 중간에 변경되지 않고 이전 사이클과 동일하게 유지($stable)되어야 합니다.

3. Timeout / Address Range

property p_timeout;
  req |-> ##[1:32] ack;
endproperty

Deadlock (교착 상태) 방지 시스템에 요청(Req)이 발생한 후 무한정 대기하는 것을 막기 위해, 최대 허용 사이클(예: 32 사이클) 내에 반드시 응답(Ack)이 수신됨을 보장합니다. 시스템 Hang을 검출하는 핵심 패턴입니다.

6 Action Block, UVM 연동 및 디버깅

Assertion이 실패(Fail)했을 때 단순히 터미널에 로그만 남기는 것을 넘어, UVM(Universal Verification Methodology) 환경과 적극적으로 연동하는 것이 실무적인 Best Practice입니다.

else 블록(Action Block)을 활용하여 에러 발생 시 커스텀 메시지를 출력하고, 시뮬레이션을 중단시키거나 커버리지 데이터를 수집할 수 있습니다. 

// 15. Reset Assertion (disable iff)
ASSERT_RV_STABLE: assert property (
    @(posedge clk) disable iff (!rst_n)
    p_rv_stable
) // 19. Action Block (Pass)
else begin // Action Block (Fail)
    // 20. UVM 연동
    `uvm_error("SVA_ERR", 
        $sformatf("Stable violated! data=%h", data))
end
  • Assertion Labeling: ASSERT_RV_STABLE처럼 명확한 레이블(이름)을 부여해야 Verdi와 같은 Waveform Debug 툴에서 해당 에러 발생 시점으로 즉시 이동(Trace)하기 용이합니다.
  • Reset 필터링 (disable iff): 시스템 초기화(Reset) 구간에서는 신호가 불안정하므로 disable iff (!rst_n)를 추가하여 False Alarm(오탐)을 원천 차단해야 합니다.
  • UVM Error 매핑: `uvm_error 매크로를 호출하면 UVM 리포팅 시스템이 이를 감지하여 최종 테스트 결과를 FAIL로 마킹하고, Regression 관리 도구에 정확하게 보고합니다.

7 고급 시퀀스 결합 연산자 (Sequence Composition)

복잡한 프로토콜을 검증할 때는 여러 개의 시퀀스를 시간 축에서 겹치거나 병합해야 할 때가 있습니다. SVA는 여러 시퀀스의 시간적 관계를 명확히 정의하는 결합 연산자를 제공합니다.

seq1 intersect seq2

두 시퀀스가 정확히 동일한 시점에 시작하고 종료해야 참(True)이 됩니다.

seq1 within seq2

seq1이 seq2의 시작과 종료 범위(Window) 안에서 완전히 발생해야 합니다.

expr throughout seq

시퀀스가 진행되는 전체 기간 내내 특정 조건(expr)이 참으로 유지되어야 합니다.

8 비동기 리셋 처리와 평가 취소 (disable iff)

하드웨어 동작 중 리셋(Reset)이 인가되면 진행 중이던 모든 트랜잭션은 즉각 폐기되어야 합니다. SVA도 마찬가지로 프로퍼티 평가 도중 리셋이 떨어지면, 이를 Assertion Fail로 처리하는 것이 아니라 평가 취소(Abort/Kill)로 안전하게 무시해야 합니다.

이를 위해 SVA에서는 disable iff 지시어를 지원하며, 이는 클럭 엣지에 동기화되지 않고 비동기적으로(Asynchronously) 동작하여 즉시 평가를 멈춥니다. 

assert property (
    @(posedge clk) 
    disable iff (!rst_n) // rst_n이 0이 되면 즉시 취소
    req |-> ##[1:5] ack
);

9 Assertion 삽입 위치와 무결점 플러그인 (bind)

SVA 코드는 물리적으로 어디에 작성해야 할까요? 설계 코드(RTL) 안에 직접 섞어 쓰는 방식은 설계와 검증의 분리(Separation of Concerns) 원칙에 위배됩니다. 실무에서는 인터페이스 내부나 별도의 Checker 모듈에 SVA를 작성하고, bind 지시어를 통해 원격으로 연결하는 방식을 표준으로 사용합니다.

왜 bind를 사용해야 할까요?

  • RTL 무결성 보존: 검증 코드를 넣기 위해 원본 설계 코드를 수정할 필요가 없습니다. (Risk Zero)
  • 검증 환경의 유연성: 합성(Synthesis) 시에는 검증 모듈을 빼고, 시뮬레이션 환경(Testbench) 최상단에서만 결합하여 오버헤드를 없앨 수 있습니다.
  • Third-party IP 검증: 수정이 불가능하게 암호화되거나 미리 컴파일된 외부 IP 모듈 내부의 신호에도 Assertion 모듈을 플러그인할 수 있습니다.

Bind 문법 및 사용 예시

// 1. 순수 Assertion Module 정의
module apb_checker (input pclk, prst_n, psel);
    // ... SVA properties & assert ...
endmodule

// 2. Testbench 상단에서 DUT에 플러그인 (Bind)
// 문법: bind [Target] [Checker] [Instance_name] (Ports)
bind apb_slave_dut apb_checker u_chk (
    .pclk(clk_i),      // DUT 내부 신호 맵핑
    .prst_n(rst_ni),
    .psel(sel_i)
);
