UVM Architecture Guide

Modeling a Bus Using a Class

검증 환경에서 버스(Bus) 프로토콜이나 트랜잭션 데이터를 효과적으로 생성하고 전달하기 위해 uvm_sequence_item을 상속받아 클래스로 모델링하는 방법을 다룹니다.

1

Item의 역할

Sequence Item은 검증 환경 내에서 데이터를 운반하는 기본 단위입니다. 마치 택배 상자처럼 주소(Address), 데이터(Data), 제어 신호(Control) 등을 캡슐화하여 생성기(Sequence)에서 구동기(Driver)로 전달합니다.

  • 데이터 추상화: 복잡한 핀 레벨 신호를 논리적인 객체로 변환
  • 무작위화(Randomization): 다양한 테스트 시나리오를 위한 랜덤 데이터 생성 보장
2

상속 구조 (Inheritance)

uvm_sequence_item 심층 해부 (Anatomy)

uvm_sequence_item은 단순한 데이터 컨테이너가 아닙니다. 네트워크 통신의 OSI 7계층 모델처럼, UVM 상속 계층을 거치며 각 부모 클래스로부터 강력한 메타데이터(Header)를 누적하여 상속받습니다. 사용자가 정의한 데이터(Payload)는 이 거대한 관리 정보들로 둘러싸여 안전하게 추적 및 라우팅됩니다.

L1: uvm_object (식별 속성)

모든 UVM 객체의 최상위 부모입니다. Factory에서 객체를 생성하고 관리하기 위한 고유 인스턴스 ID(m_inst_id)와 문자열 이름(m_name)을 제공하여 시뮬레이션 전체에서 객체를 식별 가능하게 합니다.

L2: uvm_transaction (시간/이벤트)

트랜잭션 레벨 모델링(TLM)을 위한 계층입니다. 파형(Waveform) 뷰어에 트랜잭션을 박스 형태로 시각화하기 위해 트랜잭션의 생성/종료 시간(begin_time, end_time)과 고유 트랜잭션 ID를 관리합니다.

L3: uvm_sequence_item (라우팅)

  • sequence_id: 여러 시퀀스가 동시 동작할 때, Driver가 응답(Response)을 정확히 원래 요청자에게 반환하도록 돕는 라우팅 키입니다.
  • sequencer_ptr: 이 아이템을 처리하고 있는 대상 Sequencer의 포인터(핸들)를 저장하여 상태 제어에 사용됩니다.

L4: my_bus_item (사용자 페이로드)

설계자가 직접 정의하는 영역입니다. 하드웨어 핀으로 구동될 addr, data 값들이 담겨 있으며, 위의 3개 레이어(L1~L3)가 제공하는 메타데이터의 보호와 안내를 받아 Driver까지 안전하게 전달됩니다.

Item 클래스 내부 구조 설계

성공적인 Bus Modeling을 위해 필수적으로 포함되어야 하는 요소들입니다.

[3] Field Modeling

버스 프로토콜에 필요한 신호(Address, Data, Command 등)를 클래스의 변수로 정의합니다. 시퀀스에서 자동 생성을 위해 rand 키워드를 사용합니다.

[5] Factory & [6] Automation

`uvm_object_utils 매크로를 통해 UVM Factory에 객체를 등록합니다. `uvm_field_* 매크로는 print, copy, compare 등의 기본 함수를 자동화합니다.

[4] Constructor (new)

UVM Object의 생성자는 보통 하나의 인자(name)만을 받습니다. 내부에서 부모 클래스의 생성자인 super.new(name)을 반드시 호출해야 합니다.

Field Automation: `uvm_field_* 매크로의 마법

객체 지향 프로그래밍에서 클래스 내부에 선언된 변수들을 제대로 활용하려면 깊은 복사(Deep Copy), 문자열 출력(Print), 두 객체 간의 값 비교(Compare) 등의 코드를 일일이 작성해야 합니다. UVM에서는 `uvm_field_* 매크로를 통해 이러한 지루하고 반복적인 수백 줄의 코드를 단 한 줄로 자동화(Automation)합니다.

print()

시뮬레이션 로그 파일에 객체 내부의 모든 변수 이름, 타입, 크기, 현재 할당된 값을 표 형태(Table format)로 깔끔하게 정렬하여 출력합니다.

copy()

원본 객체와 동일한 데이터를 가지는 새로운 객체를 생성합니다. 포인터 주소만 넘기는 얕은 복사(Shallow copy)가 아닌 완벽한 깊은 복사(Deep copy)를 지원합니다.

compare()

Scoreboard 등에서 DUT의 출력 결과와 참조 모델(Reference Model)의 결과가 일치하는지 변수 단위로 자동 비교하고, 불일치 시 에러 로그를 띄워줍니다.

Automation Flags: 세밀한 동작 제어

`uvm_field_* 매크로의 두 번째 인자로 들어가는 플래그(Flag)들은 비트 논리합 연산자(|)를 사용하여 조합됩니다. 기본 묶음(Preset)을 바탕으로, 특정 변수의 출력 형식을 바꾸거나 비교/복사 대상에서 제외하는 등 동작을 커스터마이징할 수 있습니다.

1. 동작 제어 (Action Control)

각 자동화 기능의 활성화/비활성화 여부를 명시적으로 지정합니다. (ON vs OFF)

  • UVM_COPY copy() 수행 UVM_NOCOPY
  • UVM_COMPARE compare() 수행 UVM_NOCOMPARE
  • UVM_PRINT print() 수행 UVM_NOPRINT
  • UVM_RECORD Waveform 기록 UVM_NORECORD
  • UVM_PACK Bit-stream 변환 UVM_NOPACK
  • UVM_UNPACK Bit-stream 복원 UVM_NOUNPACK
  • UVM_SET Config DB 설정
    UVM_NOSET UVM_READONLY

2. 묶음 (Presets / Bundles)

UVM_ALL_ON UVM_DEFAULT UVM_FLAGS_ON UVM_FLAGS_OFF

UVM_ALL_ON 또는 UVM_DEFAULT는 모든 기능을 활성화하는 가장 일반적인 베이스 플래그입니다.

3. Object 복사 정책 (Copy Policy)

클래스 내부의 객체(Object 핸들)를 복사할 때의 정책입니다.

UVM_DEFAULT_POLICY UVM_DEEP UVM_SHALLOW UVM_REFERENCE

4. 출력 형식 (Radix / Format)

print() 함수가 호출될 때 시뮬레이션 로그에 값이 표시되는 진수나 데이터 포맷을 결정합니다.

UVM_BIN (2진수) UVM_OCT (8진수) UVM_DEC (10진수) UVM_HEX (16진수) UVM_UNSIGNED UVM_TIME UVM_REAL UVM_REAL_DEC UVM_REAL_EXP UVM_STRING UVM_ENUM UVM_UNFORMAT2 UVM_UNFORMAT4 UVM_NORADIX

5. 호환 / 특수 (Special / Advanced)

UVM_PHYSICAL 실제 하드웨어 연결용 필드 표시
UVM_ABSTRACT 추상화된 가상 필드 표시
UVM_NODEFPRINT Default Print에서 제외
7

Constraint (제약조건)

rand 변수가 가질 수 있는 유효한 값의 범위를 지정합니다. 유효하지 않은 주소 접근을 막거나, 특정 데이터 패턴을 집중적으로 테스트하기 위해 필수적입니다.

constraint addr_align_c {
  addr[1:0] == 2'b00; // 4-byte align
}
8

Optional Hooks

무작위화 전후에 자동으로 실행되는 콜백 함수입니다. 무작위화 불가능한 필드(예: Parity bit)를 생성된 데이터를 바탕으로 계산할 때 유용합니다.

[9] 실제 코드 구성 흐름 & [10] 실행 흐름

Execution Flow: Sequence → Driver → DUT

1. Sequence (생성)

my_bus_item 객체를 생성(create)하고, 제약조건에 맞춰 무작위화(randomize)한 뒤 Driver로 전송합니다.

2. Driver (변환)

포트를 통해 Item을 수신합니다. 클래스 내의 addr, data 값을 실제 하드웨어 Interface 핀 신호로 구동(Driving)합니다.

3. DUT (동작)

Driver가 구동한 전기적 신호(0과 1)를 입력받아 설계된 RTL 로직에 따라 동작을 수행합니다.
