UVM 기초 라이브러리

3. UVM Core Mechanisms

검증 방법론의 뼈대를 이루는 핵심 클래스와 동적 객체 생성, 계층 간 데이터 전달, 시뮬레이션 위상 동기화의 원리를 시각적으로 완벽하게 분해합니다.

[1]~[3]

UVM 클래스 계층 구조

UVM의 최상위 추상 클래스부터 동적 객체와 정적 컴포넌트까지 이어지는 상속 흐름을 한 화면에서 정리합니다.

다이어그램 탐색

좌측 다이어그램에서 클래스 블록을 클릭하면 프레임워크 내 역할과 핵심 기능을 확인할 수 있습니다.

uvm_void

가상 추상화의 시작점

SystemVerilog는 C++처럼 다중 상속(Multiple Inheritance)이나 최상위 기본 클래스를 강제하지 않습니다. 이를 극복하기 위해 UVM 프레임워크가 인위적으로 만든 아무런 속성도, 메서드도 없는 텅 빈 추상 클래스(Abstract Root Class)입니다.

존재 목적

다형성(Polymorphism) 구현 시, 어떠한 UVM 객체라도 담을 수 있는 보편적인 컨테이너(Generic Handle) 역할을 수행하기 위해 존재합니다.

uvm_object

UVM의 동적 데이터 생명체

메모리상에 동적(Dynamic)으로 생성되고 소멸하는 데이터 구조의 기본 클래스입니다. 시뮬레이션 중 수천, 수만 개가 생성될 수 있으며 Factory 엔진을 통해 객체의 치환(Override)이 가능합니다.

핵심 특징 & 메서드

  • 데이터 관리 내장: copy(), clone(), compare(), print()
  • 정적인 계층 구조(Hierarchy)나 부모/자식 관계를 가지지 않습니다.
uvm_transaction

통신 패킷의 추상화

uvm_object에서 파생되어, 검증 환경 내부에서 컴포넌트 간 데이터를 주고받을 때 사용되는 단위 트랜잭션(Packet)입니다.

  • 타이밍 및 로깅 관리: 생성 시간(Accept time)과 시작/종료 시간 기록 기능 포함.
  • 현업에서는 이 클래스를 한 번 더 감싼 uvm_sequence_item을 상속받아 DUT로 보낼 시그널 데이터(Payload)를 정의합니다.
uvm_report_object

전역 메시징 처리기

uvm_object에 **UVM 리포팅 시스템(메시지 출력, 필터링, 로깅)** 기능을 추가한 클래스입니다. 시뮬레이션 중 발생하는 모든 상태와 오류를 표준화된 포맷으로 출력할 수 있게 해줍니다.

  • 📝 제공 매크로: `uvm_info, `uvm_warning, `uvm_error
  • ⚙️ 제어: 메시지의 심각도(Severity)와 출력 레벨(Verbosity)을 조정하여, 화면에 표시할 로그를 세밀하게 필터링할 수 있습니다.
uvm_component

검증 환경의 정적 구조물

시뮬레이션 시작 전(Build)에 생성되어 끝날 때까지 메모리에 정적(Static)으로 고정되어 있는 검증 환경의 뼈대입니다. uvm_report_object를 상속받습니다.

핵심 특징 (객체와의 차별점)

  • 계층(Hierarchy) 형성: new(name, parent) 인자를 통해 트리 형태의 부모-자식 구조를 가집니다.
  • Phase 메커니즘 탑재: build, connect, run 등 정해진 시간에만 실행되는 위상 시스템을 가집니다.
  • Test, Env, Agent, Driver 등 하드웨어 블록에 대응하는 요소입니다.
[4]

The UVM Factory Core Engine

타입 등록, Override 규칙, 생성 요청 해석 과정을 Factory 중심으로 묶어 UVM의 동적 생성 구조를 설명합니다.

  • 🔀

    제어 역전 (IoC) : 객체의 생성(new)을 개발자가 직접 하지 않고 Factory가 대행하여 프레임워크가 검증 환경 구성의 주도권을 가집니다.

  • 🎭

    다형성 (Polymorphism) : 원본 소스 코드 수정 없이 런타임에 원하는 파생 클래스로 객체를 치환(Override)하여 다양한 테스트 시나리오를 주입합니다.

  • 🌐

    전역 레지스트리 : 모든 UVM 컴포넌트와 트랜잭션의 타입을 중앙에서 통합 관리하는 싱글톤(Singleton) 데이터베이스 역할을 수행합니다.

`uvm_component_utils Expansion 
// 정적 구조물(Env, Agent 등)을 위한 매크로 정의
`define uvm_component_utils(T) \
  /* 컴포넌트용 전용 Registry 타입 사용 */ \
  typedef uvm_component_registry #(T, `"T`") type_id; \
  \
  static function type_id get_type(); \
    return type_id::get(); \
  endfunction \
  \
  virtual function uvm_object_wrapper get_object_type(); \
    return type_id::get(); \
  endfunction
`uvm_object_utils Expansion 
// 동적 데이터(Transaction, Config 등)를 위한 매크로 정의
`define uvm_object_utils(T) \
  /* 객체용 전용 Registry 타입 사용 */ \
  typedef uvm_object_registry #(T, `"T`") type_id; \
  \
  static function type_id get_type(); \
    return type_id::get(); \
  endfunction \
  \
  virtual function uvm_object_wrapper get_object_type(); \
    return type_id::get(); \
  endfunction

[6] type_id::create()

✨

객체 생성 의뢰 (Delegation)

  • 호출 방식: SystemVerilog의 기본 생성자인 new()를 직접 호출하지 않고, Factory에게 간접적으로 객체 생성을 의뢰함.
  • 핵심 용도: 다형성(Polymorphism)과 Override 메커니즘을 적용하기 위한 필수 전제 조건임.
  • 반환 결과: Factory가 타입과 설정된 규칙을 검사한 후, 가장 적합한 인스턴스를 생성하여 반환함.
drv = my_driver::type_id::create("drv", this);

new() vs type_id::create() 전격 비교

SystemVerilog 직접 생성 Hardcoded

  • ✖ 정적 결합 (Tight Coupling): 생성할 객체의 타입이 컴파일 시점에 고정되어 유연성이 전혀 없습니다.
  • ✖ Override 불가: 에러가 있는 드라이버를 다른 드라이버로 교체하려면 환경(Env) 소스 코드를 직접 수정하고 재컴파일해야 합니다.

UVM Factory 위임 생성 Dynamic

  • ✔ 동적 결합 (Loose Coupling): 런타임에 팩토리가 어떤 타입의 객체를 반환할지 결정하므로 매우 유연합니다.
  • ✔ Override 핵심: 원본 환경 코드(Env)를 단 한 줄도 건드리지 않고, 테스트(Test) 단계에서 객체를 다른 객체로 치환할 수 있습니다.

🔄

동적 타입 치환 (Type Override)

  • 주요 목적: 기존 소스 코드(RTL/TB)의 일절 수정 없이, 테스트 시나리오의 목적에 따라 생성될 객체의 타입을 동적으로 변경함.
  • 동작 원리: 환경 코드(Env)는 원본 클래스 생성을 요청하지만, Factory가 이를 가로채어 파생 클래스(my_driver_ext)를 대신 생성하고 주입(Inject)함.
  • 지원 방식: 전역적으로 변경하는 타입 기반 치환 및 특정 경로만 변경하는 인스턴스 기반 치환을 완벽히 지원함.
factory.set_type_override_by_type(
  my_driver::get_type(), 
  my_driver_ext::get_type()
);

[8] uvm_config_db

컴포넌트 간 물리적인 포트 연결 없이, 계층(Hierarchy)의 문자열 경로를 기반으로 설정값, 객체, 가상 인터페이스 등을 저장하고 검색하는 중앙 집중형 데이터베이스입니다.

set(context, inst_name, field_name, value)

상위 계층(보통 Top 모듈이나 Test)에서 하위 컴포넌트들이 사용할 데이터를 DB에 저장합니다. inst_name에 정규표현식(예: *)을 사용하여 여러 컴포넌트에 일괄 적용 가능합니다.

uvm_config_db#(virtual intf)::set(null, "uvm_test_top.env.*", "vif", top_vif);

get(context, inst_name, field_name, value)

하위 계층(Driver, Monitor 등)에서 build_phase 중에 자신에게 할당된 데이터를 DB에서 꺼내옵니다. 문자열 매칭이 정확히 일치해야 합니다.

uvm_config_db#(virtual intf)::get(this, "", "vif", my_vif);

타입 매칭이 엄격합니다. 파라미터 #(type)이 정확히 일치하지 않으면 get()에 실패합니다.

내부 레지스트리 검색 메커니즘

set() 함수가 호출되면 데이터는 내부적으로 문자열 기반의 해시 테이블(Hash Table)에 저장됩니다. 이후 하위 컴포넌트에서 get() 함수가 호출될 때 자신의 계층 경로(Hierarchy Path)와 정확히 일치하거나 정규표현식(Wildcard)에 매칭되는 데이터를 찾아 반환합니다.

💡

문자열(String) 기반의 계층 검색 방식

  • 설계 유연성: 컴포넌트 간의 물리적인 연결(포인터나 포트) 없이 문자열 경로만으로 전역 데이터를 공유할 수 있습니다.
  • 정규표현식 매칭(*): 상위 환경(Env)에서 "*.env.*" 처럼 와일드카드를 지정하면, 조건을 만족하는 하위의 여러 에이전트가 단일 set()만으로도 동일한 인터페이스를 공유받게 됩니다.
  • 우선순위 제어: 동일한 경로와 필드명을 가진 값이 중복으로 저장될 경우, 시뮬레이션 계층이 높은 곳(예: Top Module)에서 설정한 값이 가장 높은 우선순위를 가집니다.

UVM 동기화 및 제어 메커니즘

검증 환경의 모든 컴포넌트가 동일한 생명 주기를 갖도록 실행 상태를 일치시키는 Phase 메커니즘과, 시뮬레이션의 동적 종료 시점을 제어하는 Objection 시스템의 아키텍처를 분석합니다.

[9] uvm_phase (실행 위상 동기화)

UVM Phase는 검증 환경을 구성하는 수많은 인스턴스들이 정해진 절차에 따라 순차적으로 동작하도록 제어하는 전역 스케줄링 프레임워크입니다. Top 모듈에서 run_test()가 호출되면, 내부의 Phase Scheduler가 작동하여 아래의 9가지 주요 단계를 순차적으로 파생시킵니다. 오직 run_phase만이 실제 시뮬레이션 시간을 소모(Time-Consuming Task)하며, 나머지 위상은 즉각적으로 처리되는 함수(Function) 형태로 실행됩니다.

1

Top-Down 위상 구조

build_phase가 대표적입니다. 최상위 모듈부터 하위 컴포넌트 순서로 객체를 인스턴스화하며, 시뮬레이션 시간을 소모하지 않는(0ns) function 형태로 정적 실행됩니다.

2

Bottom-Up 위상 구조

connect_phase 및 검사/보고 위상입니다. 트리 구조의 최하단(Leaf) 컴포넌트부터 상위로 올라가며 포트 연결이나 시뮬레이션 결과 취합 연산을 수행합니다.

3

병렬 처리(Task) 위상

run_phase는 유일한 task 위상입니다. 모든 컴포넌트가 fork-join_none 방식으로 동시에 독립 실행을 시작하며, 클럭 지연 등 동적 시뮬레이션 시간이 소비됩니다.

[10] Objection 제어 시스템

UVM 프레임워크는 run_phase 내에 수많은 병렬 프로세스(fork-join_none)가 존재하기 때문에, 시뮬레이션을 언제 종료해야 할지 알 수 없습니다. 이를 해결하기 위해 각 컴포넌트가 부여받은 '종료 거부권(Veto)'을 행사하는 투표 시스템이 Objection 메커니즘입니다.

raise_objection()

Veto (거부권 행사)

"내 작업이 안 끝났으니 시뮬레이션을 끝내지 마라!"
시퀀스(Sequence)가 아이템 생성을 시작하거나, 드라이버가 버스 통신을 시작하는 등 Time-consuming 작업 진입 시 전역 카운터를 +1 증가시켜 강제 종료를 막습니다.

drop_objection()

Release (투표권 회수)

"내 임무는 완수했으니, 이제 끝내도 좋다."
할당된 패킷 전송이 완료되어 유휴 상태(Idle)로 돌아갈 때 호출합니다. 투표권을 회수하여 카운터를 -1 감소시킵니다.

카운터 0의 의미 (All Dropped)

모든 컴포넌트의 Objection 합계가 0이 되는 즉시 남은 스레드의 상태와 무관하게 run_phase는 즉각 종료(Kill)됩니다. 따라서 너무 일찍 drop()하면 남은 패킷이 버스에 전달되지 못하고 시뮬레이션이 잘리는 문제가 발생할 수 있습니다.

[12] TLM 통신

Transaction Level Modeling. 핀(Pin) 단위의 복잡한 시그널 토글링 대신, 객체(Packet/Transaction) 단위로 컴포넌트 간 데이터를 주고받아 시뮬레이션 속도를 높입니다.

[13] 동기화 객체

다수의 프로세스(fork-join)가 실행될 때, 실행 순서를 맞추거나 공유 자원의 충돌을 방지하기 위해 사용하는 특수 목적 클래스들입니다.

⚡ uvm_event

임의의 시점에 이벤트를 발생(trigger)시키고, 다른 곳에서 이를 대기(wait_trigger)합니다. 프로세스 간 상태 신호 전달에 쓰입니다.

🚧 uvm_barrier

지정된 개수(Threshold)의 다수 프로세스가 모두 도달할 때까지 차단(Block)시켰다가, 목표 수에 도달하면 동시에 실행을 재개시킵니다.

🔑 semaphore

한정된 자원에 여러 프로세스가 접근할 때 충돌을 막는 Mutex입니다. 열쇠(Key)를 획득(get)한 프로세스만 자원을 독점 사용할 수 있습니다.

[11] Reporting System

메시지의 심각도(Severity)와 상세 수준(Verbosity)을 기반으로 로그 출력을 제어합니다. 시뮬레이션 속도 저하를 막기 위해 불필요한 로그는 필터링할 수 있습니다.
