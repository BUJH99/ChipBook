[1] Driver의 역할 & [3] Active Agent

Active Agent는 시뮬레이션에서 '구동(Drive)'과 '모니터링(Monitor)'을 동시에 수행하는 에이전트입니다. 이 안에서 Driver는 Sequencer로부터 받은 상위 레벨의 Transaction(예: AXI/APB Read/Write 패킷)을 해석하여, 지정된 프로토콜 스펙과 클럭(Clock) 타이밍에 맞게 실제 하드웨어의 물리적 핀 신호(0과 1)로 변환해 DUT에 인가(Pin Wiggle)합니다.

💡 Key Insight: Transaction은 "주소 0x100에 데이터 0xFF를 써라"라는 S/W적 지시라면, Driver는 이를 "Clock Rising Edge에서 PADDR=0x100, PWRITE=1, PWDATA=0xFF로 1사이클 유지하라"는 H/W적 동작으로 번역하는 통역사 역할을 합니다.

[2] uvm_driver 상속구조

직접 설계하는 사용자 드라이버는 반드시 uvm_driver를 상속받아야 합니다. 이때, #(REQ, RSP) 파라미터를 통해 Sequencer와 주고받을 Transaction의 타입을 명시합니다. (일반적으로 RSP는 생략하여 REQ와 동일하게 사용합니다.)

class my_driver extends uvm_driver #(my_transaction);

[4] Interface vs Virtual Interface

동적으로 메모리에 할당되는 Class(S/W)는 하드웨어 배선 덩어리인 정적 Module(H/W) Interface를 직접 가질 수 없습니다. 따라서 virtual interface라는 '포인터(핸들)'를 통해 물리적 선들에 접근합니다. 이 둘을 이어주는 매개체가 바로 uvm_config_db입니다.

[5] 연결 타이밍 & [6] build_phase & [7] run_phase

Driver의 생명주기는 철저히 분리됩니다. 준비 단계인 build_phase에서 하드웨어와 연결할 포인터를 미리 설정하고, 실제 시뮬레이션 시간이 흐르는 run_phase에서는 무한 루프를 돌며 신호를 지속적으로 구동합니다.

// [6] build_phase: Get Virtual Interface
function void build_phase(uvm_phase phase);
  super.build_phase(phase);
  // 하위 컴포넌트(Driver)는 값을 받아옵니다(get)
  if (!uvm_config_db#(virtual bus_if)::get(this, "", "vif", vif))
    `uvm_fatal("NO_VIF", "Virtual interface not found")
endfunction
// [7] run_phase: Active Loop
task run_phase(uvm_phase phase);
  // 시뮬레이션 내내 종료되지 않는 무한 루프
  forever begin
    seq_item_port.get_next_item(req); // 1. Wait & Get
    drive_one_item(req); // 2. Drive to DUT
    seq_item_port.item_done(); // 3. Handshake Done
  end
endtask

Driver Transaction Loop 메커니즘

Sequencer와 Driver 사이의 통신은 매우 엄격한 Handshake 프로토콜을 따릅니다. 요청(Req)을 받아 처리하고, 완료(Done)를 알리기 전까지 다음 Transaction은 전달되지 않습니다.

[8] seq_item_port

Driver 내부에 내장된(built-in) TLM 포트입니다. Sequencer의 seq_item_export와 연결(Connect)되어 두 컴포넌트 간의 고속도로 역할을 합니다.

[9] get_next_item()

Sequencer 내부 FIFO에 새로운 Transaction이 들어올 때까지 대기(Blocking)합니다. 아이템이 준비되면 이를 가져와 루프의 다음 단계를 진행합니다.

[11] drive_one_item()

사용자가 직접 구현하는 User Task입니다. `req` 객체의 데이터를 Virtual Interface의 핀에 인가하며, 클럭 지연(`@(posedge clk)`) 등 실제 시뮬레이션 타임을 소모합니다.

[10] item_done()

하드웨어 구동이 완벽히 끝났음을 Sequencer에 알립니다. 이 함수가 호출되어야만 Sequencer가 블로킹을 풀고 다음 Transaction을 보낼 수 있습니다.

[12] HW Rule 준수 및 Signal Control Task

실무에서는 drive_one_item() 내에 모든 로직을 하드코딩하지 않습니다. 대신, 프로토콜 스펙(APB, AXI 등)의 엄격한 HW 규칙(Setup/Hold 시간, Handshake, Burst 등)을 지키기 위해 개별 Task(예: drive_write, drive_read)로 분리하여 핀 레벨의 정밀한 신호 제어(Signal Control)를 수행합니다.

// Dispatcher: 명령어에 따른 Task 분기
task drive_one_item(my_tx req);
  case (req.cmd)
    WRITE: drive_write(req);
    READ : drive_read(req);
  endcase
endtask
// HW Rules: 정밀한 Signal 제어 및 Handshake
task drive_write(my_tx req);
  // 1. Clock Sync (Setup time 보장)
  @(posedge vif.clk);

  // 2. Drive Data & Valid High
  vif.valid <= 1'b1;
  vif.data <= req.data;

  // 3. Wait for Handshake (HW Rule)
  wait(vif.ready == 1'b1);

  // 4. Clear signals after drive
  @(posedge vif.clk);
  vif.valid <= 1'b0;
endtask
