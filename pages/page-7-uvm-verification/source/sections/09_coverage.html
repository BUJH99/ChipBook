Verification Guide

Coverage : Check Verification Completeness

검증이 얼마나 완벽하게 이루어졌는지 정량적으로 측정하는 Coverage의 역할과 UVM 환경에서의 구현 구조를 시각적으로 이해합니다.

1 Coverage의 역할

디자인의 모든 기능이 제대로 테스트되었는지 확인하는 지표(Metric)입니다. "검증을 언제 끝낼 수 있는가?"에 대한 명확한 해답을 제공합니다.

  • 테스트 진행 상황의 정량적 수치화 (0% ~ 100%)
  • 미테스트 영역(Coverage Hole) 발견
  • 검증 종료 조건(Sign-off)의 기준점
Verification Progress 100% Goal
Stimulus Monitor Coverage Collection Sign-off

UVM Expert Insight : Code Coverage vs Functional Coverage

업계에서는 Code Coverage(Statement, Branch, Toggle 등 RTL 코드가 얼마나 실행되었는가)와 Functional Coverage(설계 스펙/의도대로 기능이 동작했는가)를 구분합니다. Code Coverage가 100%라고 해서 스펙을 완벽히 구현했다는 보장은 없습니다. (구현 자체가 누락된 코드는 측정되지 않기 때문입니다). 따라서 UVM 환경에서는 검증 엔지니어가 직접 스펙을 기반으로 정의한 Functional Coverage를 핵심 Sign-off 기준으로 삼습니다.

2 UVM 상속구조 & 연결 (Monitor → Coverage)

Coverage Component는 uvm_subscriber를 상속받으며, Monitor로부터 TLM Port를 통해 트랜잭션을 수신합니다.

UVM Expert Insight : 왜 uvm_subscriber를 사용할까요?

UVM의 핵심 철학은 '컴포넌트 간의 결합도(Coupling)를 낮추는 것'입니다. Monitor는 트랜잭션을 수집하여 analysis_port (ap)를 통해 단순히 브로드캐스트할 뿐, 누가 그 데이터를 사용하는지 알지 못합니다. Coverage Component는 uvm_subscriber를 상속받아 포트를 연결하고 데이터를 수동적으로 받아 처리합니다. 이렇게 분리하면 Monitor 코드를 수정하지 않고도 Scoreboard나 Coverage Collector를 자유롭게 붙이거나 뗄 수 있어 재사용성이 극대화됩니다.

3 Functional Coverage 구조 (Covergroup ~ Cross)

테스트 목표를 정의하는 covergroup과 트랜잭션이 Bins에 매핑되어 Cross 매트릭스를 채우는 흐름입니다.

💡 UVM Expert Insight : Bins와 Coverage Hole

Coverpoint 내부의 각각의 bins가 모두 채워져야 해당 항목이 100%가 됩니다. 위 그림의 매트릭스에서 0번 찍힌 붉은색 영역(Coverage Hole)은 (High Address + Write Cmd) 조합이 시뮬레이션 중 단 한 번도 발생하지 않았음을 의미합니다.

이 Hole을 메우기 위해 검증 엔지니어는 시드(Seed)를 바꾸거나 테스트 벤치의 Constrained Random 구문을 수정하여 해당 시나리오로 Stimulus를 유도해야 합니다.

⚠️ State Explosion과 예외 처리 (ignore_bins)

단순히 모든 변수를 Cross하면 상태 공간 폭발(State Space Explosion)이 발생하여 툴에 엄청난 부하를 줍니다.

만약 저 'Coverage Hole'이 "스펙상 절대 발생할 수 없는 조합"(예: Read-only 영역에 Write 시도)이라면, 해당 조합을 ignore_bins나 illegal_bins로 지정하여 전체 모수(Denominator)에서 제외해야 비로소 최종 Coverage 100% Goal을 달성할 수 있습니다.

4 [10] write() Function

Subscriber의 핵심 콜백(Callback) 함수입니다. Monitor로부터 트랜잭션을 수신할 때마다 즉시 호출되어 Coverage를 샘플링합니다.

virtual function void write(T t);
  // 1. 트랜잭션 데이터 복사
  this.tr = t;

  // 2. Covergroup 샘플링 트리거
  this.cg.sample();
endfunction

UVM Expert Insight : Zero-Time Execution

SystemVerilog에서 function은 시간 지연(#10, @(posedge clk) 등)을 허용하지 않는 즉시 실행 구문입니다. Monitor가 버스에서 패킷을 캡처하여 ap.write()를 호출하는 순간, 연결된 모든 Subscriber의 write() 콜백이 시뮬레이션 타임(Tick) 증가 없이 Zero-Time으로 동기화되어 실행됩니다. 이를 통해 트랜잭션의 유실이나 타이밍 왜곡 없이 가장 정확한 상태에서 Coverage를 수집할 수 있습니다.

5 [11] Reset Handling

리셋 구간에서는 잘못된 값이 수집되지 않도록 stop(), 이후 start()를 제어하여 데이터 오염을 방지합니다.

UVM Expert Insight : False Positive 방어선

리셋이 인가되는 동안 DUT 내부 레지스터는 강제로 0으로 초기화되거나 X(Unknown) 상태가 될 수 있습니다. 만약 이 Garbage Data가 Monitor를 거쳐 Subscriber로 전달될 때 cg.stop()으로 막지 않는다면, Address=0, Cmd=0 조합의 Coverage Bin이 달성된 것으로 잘못 기록됩니다. 이를 방지하기 위해 iff(!reset) 조건식과 stop()/start() 메소드를 이중 방어선으로 반드시 구축해야 합니다.

6 [12] Coverage Report & Regression Merge

단일 시뮬레이션(Single Test)만으로는 복잡한 디자인의 모든 코너 케이스를 커버할 수 없습니다. 실무에서는 수백 개의 서로 다른 Random Seed를 사용하여 시뮬레이션을 반복(Regression)하고, 여기서 생성된 Coverage Database들을 하나로 병합(Merge)하여 최종 Report를 생성합니다.

UVM Expert Insight : Regression & Merge

하나의 시뮬레이션에서 100% Coverage를 달성하는 것은 불가능에 가깝습니다. 실무에서는 UVM Randomization을 활용하여 수천 개의 Random Seed로 병렬 시뮬레이션(Regression)을 돌립니다. 이후 각 결과 파일(.ucdb 등)을 Merge(병합)하여 최종 누적 Coverage를 확인하는 것이 검증의 표준 워크플로우입니다.

Sign-off와 Exclusion (Waiver)

모든 기능 Coverage가 100%에 도달해야 테이프아웃(Sign-off)이 가능합니다. 단, 스펙상 발생할 수 없는 시나리오(예: Reserved 영역 접근)로 인해 발생한 Coverage Hole은 팀 리뷰를 거쳐 Exclusion(Waiver) 처리하여 강제로 분모에서 제외시킵니다. 대시보드의 'WAIVER' 마크가 이를 의미합니다.

7 [13] 실제 코드 구조 (UVM Subscriber) 

class my_coverage extends uvm_subscriber #(my_transaction);
  `uvm_component_utils(my_coverage)

  my_transaction tr;

  // [5] covergroup 정의
  covergroup cg_bus;
    // [8] 옵션/조건 설정
    option.per_instance = 1;
    // Reset이 아닐 때만 수집
    
    // [6] coverpoint 정의 (Address)
    cp_addr: coverpoint tr.addr iff(!tr.reset) {
      // [7] bins 정의
      bins low_range  = {[0:8'h7F]};
      bins high_range = {[8'h80:8'hFF]};
    }

    // [6] coverpoint 정의 (Command)
    cp_cmd: coverpoint tr.cmd iff(!tr.reset) {
      bins read  = {2'b01};
      bins write = {2'b10};
    }

    // [9] cross Coverage
    cross_addr_cmd: cross cp_addr, cp_cmd;
  endgroup

  function new(string name, uvm_component parent);
    super.new(name, parent);
    cg_bus = new(); // covergroup 인스턴스화
  endfunction

  // [10] write function 구현 (analysis port로부터 수신)
  virtual function void write(my_transaction t);
    this.tr = t;
    
    // [11] Reset Handling 
    if (tr.reset) begin
      cg_bus.stop();  // 리셋 중 샘플링 중단
    end else begin
      cg_bus.start(); // 리셋 해제 시 샘플링 시작
      cg_bus.sample(); // Coverage 샘플링 수행
    end
  endfunction
endclass
