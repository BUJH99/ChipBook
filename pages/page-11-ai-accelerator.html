AI Accelerator

Compute Architecture

AI Accelerator는 연산기가 아니라, data reuse를 유지하는 작은 메모리 시스템입니다

AI accelerator를 설명할 때 핵심은 MAC 개수보다 weight, activation, partial sum이 어디에 머무르고 얼마나 다시 쓰이는가입니다. 좋은 NPU는 command queue, DMA, local SRAM, systolic array, quantization path가 맞물려 DRAM 왕복을 줄이고 MAC array utilization을 끌어올립니다.

Reuse / Bandwidth / Utilization / Precision
Core Architecture
MAC array는 소비자

연산기는 데이터를 먹는 쪽입니다. 충분히 먹이지 못하면 TOPS 숫자가 커도 실제 throughput은 낮습니다.

Memory Hierarchy
Local SRAM이 재사용을 만든다

activation, weight, psum buffer가 DRAM 접근을 흡수해야 전력과 latency가 줄어듭니다.

Dataflow Strategy
무엇을 고정할 것인가

weight stationary, output stationary, row stationary는 모두 트래픽을 어디서 줄일지에 대한 답입니다.

How to Read

dataflow를 설명하는 순서

1. 고정되는 값

PE 또는 local SRAM 안에서 오래 유지되는 값이 reuse의 주인공입니다.

2. 흐르는 값

계속 이동하는 activation, weight, psum이 bandwidth와 energy 비용입니다.

3. 줄어드는 traffic

DRAM 접근, psum write-back, SRAM bank conflict 중 무엇이 완화되는지 말하면 됩니다.

Interview Hook

“dataflow는 연산 방식 이름이 아니라 memory traffic 모양을 정하는 정책”이라고 잡으면, TOPS/W와 utilization까지 자연스럽게 연결됩니다.

11-3. Metric & Debug Map

성능 지표는 roofline 위에서 병목 원인을 찾는 도구입니다

TOPS/W = compute ceiling + memory ceiling + utilization loss
Debug Causality

수치가 낮을 때 보는 원인 체인

Util

MAC idle cycle이 많으면 DMA overlap, tiling, bank conflict부터 의심합니다.

BW

DRAM bandwidth가 천장이라면 dataflow나 compression, batching을 바꿔야 합니다.

Reuse

SRAM hit가 낮으면 tile 크기, loop order, weight layout이 맞지 않는 신호입니다.

Prec

INT8, BF16, mixed precision은 overflow와 golden mismatch까지 함께 검증합니다.

Metric Checklist
TOPS / TOPS/W

peak와 sustained를 구분하고 batch, precision 조건을 같이 말합니다.

Array Utilization

연산기가 놀면 memory, scheduler, sparsity 조건을 분리합니다.

SRAM Banking

bank conflict와 read/write port 수가 dataflow 한계를 만듭니다.

Verification

golden model, overflow, rounding mode, saturation을 같이 확인합니다.

One-line Summary

AI accelerator debugging은 “연산기를 더 세게”가 아니라, roofline에서 workload가 어디에 걸렸는지 보고 memory movement, scheduling, precision을 차례로 좁히는 과정입니다.
