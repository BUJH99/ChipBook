UVM / Verification

Class-Based Verification

UVM은 “테스트벤치를 큰 설계처럼 조직하는 방법”입니다

UVM은 Universal Verification Methodology의 약자로, SystemVerilog class 기반 검증 프레임워크입니다. 단순 testbench 한 파일이 아니라 transaction, sequence, driver, monitor, scoreboard, coverage를 계층적으로 나눠 대규모 IP/SoC 검증을 반복 가능하게 만드는 방식이라고 이해하면 가장 빠릅니다.

Reusable TB / Factory / TLM
Why UVM
재사용성

같은 agent, sequence, scoreboard를 IP 검증과 subsystem 검증, SoC 통합 검증에서 반복 활용하기 좋습니다.

Core View
transaction 중심

pin 토글을 바로 쓰기보다 sequence item이라는 추상 데이터 단위로 stimulus를 만들고, driver가 이를 signal로 변환합니다.

Interview Point
“언어”가 아니라 방법론

SystemVerilog 위에서 동작하는 methodology라는 점, factory override와 config_db, objection이 실무 핵심이라는 점을 같이 말하면 좋습니다.

7-1. UVM Testbench Architecture

테스트벤치 계층 구조

test → env → agent → sequencer / driver / monitor
7-2. Build to Run Phases

UVM phase는 “언제 무엇을 준비하는가”를 나누는 시간표입니다

build_phase

컴포넌트 생성. env 안에서 agent, scoreboard, coverage collector를 new/create 합니다.

connect_phase

TLM port/export 연결. monitor의 analysis port를 scoreboard, coverage에 연결하는 단계입니다.

run_phase

실제 stimulus 진행. sequence가 item을 만들고 driver가 DUT interface에 신호를 넣습니다.

extract / check / report

결과 정리, mismatch 개수, coverage 도달 여부, pass/fail 메시지를 최종 출력합니다.

7-3. Sequence Flow

stimulus는 이렇게 DUT까지 전달됩니다

1. sequence_item

addr, data, burst_len 같은 transaction payload를 담는 객체입니다.

2. sequence

랜덤화와 scenario 조합을 담당합니다. directed test + random test의 교차점으로 많이 쓰입니다.

3. sequencer

sequence와 driver 사이 arbitration 지점입니다. 어떤 item이 먼저 나갈지 조정합니다.

4. driver

transaction을 pin-level signal로 변환해 DUT interface에 넣습니다.

5. monitor / scoreboard

DUT 응답을 transaction으로 재구성하고, 기대값과 실제값을 비교해 버그를 잡습니다.

Key Object

factory / config_db

UVM의 실무 포인트는 클래스 계층만이 아니라 factory override와 uvm_config_db를 이용해 코드를 크게 바꾸지 않고 동작을 갈아끼울 수 있다는 점입니다.

override agent mode virtual interface
Key Object

objection

run_phase가 너무 빨리 끝나지 않도록 시뮬레이션 종료 타이밍을 제어합니다. raise_objection / drop_objection을 왜 쓰는지 설명할 수 있으면 이해도가 높아 보입니다.

run control test end
Key Object

analysis port / scoreboard

monitor가 뽑은 actual transaction을 scoreboard와 coverage로 fan-out하는 경로입니다. UVM에서는 이 비침투적 전달 구조가 특히 중요합니다.

TLM checker coverage
7-4. Minimal Skeleton

UVM 코드 뼈대는 이 정도 그림으로 기억하면 충분합니다

Interview-friendly View
class my_txn extends uvm_sequence_item;
  rand bit [15:0] addr;
  rand bit [31:0] data;
endclass

class my_sequence extends uvm_sequence #(my_txn);
  task body();
    my_txn req = my_txn::type_id::create("req");
    start_item(req);
    assert(req.randomize());
    finish_item(req);
  endtask
endclass

class my_driver extends uvm_driver #(my_txn);
  task run_phase(uvm_phase phase);
    forever begin
      seq_item_port.get_next_item(req);
      // req를 DUT pin-level signal로 변환
      seq_item_port.item_done();
    end
  endtask
endclass
7-5. Practical Memory Hooks

시험이나 면접에서 자주 묻는 UVM 포인트

UVM과 SV 차이

SV는 언어, UVM은 그 위의 검증 방법론입니다.

active / passive agent

active는 sequencer+driver 포함, passive는 monitor 중심으로 관찰만 합니다.

scoreboard 역할

reference model 기반 expected와 DUT actual을 비교합니다.

coverage closure

랜덤만 돌리는 것이 아니라 빈 시나리오를 채우는 방향으로 sequence를 보강합니다.
