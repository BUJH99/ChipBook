AXI Interconnect가 중앙 허브 역할을 수행합니다. 모든 주소와 데이터는 인터커넥트를 통해 직교 라우팅됩니다.

Software View
#define UART_BASE 0x40001000
volatile uint32_t* reg = UART_BASE;
*reg = 'A'; // Through AXI Fabric

소프트웨어는 I/O를 단순 메모리 접근으로 인식합니다. 인터커넥트가 물리적 주소를 찾아 신호를 라우팅합니다.

AXI + MMIO Synergy
Unified Fabric
모든 컴포넌트가 동일한 AXI 인터페이스를 사용합니다.
Address Decoding
인터커넥트가 주소에 따라 경로를 자동 분배합니다.
AXI Pipelining
I/O 접근 시에도 AXI의 throughput 이점을 그대로 활용합니다.
명령어 세트 효율성

MMIO는 CPU에서 I/O 전용 명령 로직을 제거하게 해 줍니다. LDR/STR만으로 모든 제어가 가능해져 파이프라인 구조가 더 단순해집니다.

접근 제어와 보안

AXI의 AxPROT와 MMIO를 결합하면, 특정 메모리 영역과 I/O 레지스터에 대한 보안/비보안 접근 권한을 fabric 차원에서 통합 제어할 수 있습니다.

비순차 처리 활용

I/O 장치는 메모리보다 응답이 느립니다. AXI의 ID 기반 outstanding transaction은 I/O 응답을 기다리는 동안에도 메모리 접근을 계속 흘려 throughput을 유지합니다.

Address Map 해석 예시

주소 범위는 단순한 메모리 배치가 아니라, 어떤 Slave로 트랜잭션을 라우팅할지 결정하는 하드웨어 규칙입니다.

주소 범위 선택 대상 용도
0x0000_0000 - 0x0FFF_FFFF SRAM / ROM 명령어 및 데이터의 저지연 접근 영역
0x1000_0000 - 0x1FFF_FFFF AHB Peripheral DMA 등 고속 제어 블록 연결 영역
0x4000_0000 - 0x4000_FFFF APB Peripheral UART, GPIO, Timer의 레지스터 접근 영역

실제 SoC에서는 decoder가 이 memory map에 따라 대상 Slave를 선택하므로, 주소 범위는 곧 버스 라우팅 규칙을 의미합니다.

AMBA 버스를 APB / AHB / AXI로 분화하는 이유

하나의 BUS 구조로 모든 요구사항을 만족시키기 어렵기 때문에, AMBA는 대역폭, 복잡도, 전력, 동시성 요구에 따라 프로토콜을 계층적으로 분화했습니다.

APB

레지스터 접근 중심의 저속 주변장치는 구조의 단순성, 구현 용이성, 저전력 특성이 우선되므로 APB가 적합합니다.

AHB

중간 수준의 대역폭과 공유형 시스템 버스가 필요한 구간에서는 burst와 pipeline을 지원하는 AHB 구조가 효율적입니다.

AXI

CPU, DMA, DDR처럼 높은 처리량과 동시 전송이 필요한 구간에서는 독립 채널 기반의 AXI 구조가 필요합니다.

APB (Advanced Peripheral Bus)

  • - 저전력과 설계 단순화 중심의 버스 구조
  • - 단일 전송 기반의 순차적 통신 방식

        AXI Lite Channel Architecture Diagram

        Figure 1-1 Channel architecture of AXI Lite

        AXI Lite도 5개 독립 채널을 유지하지만, 각 read/write transaction은 1-beat 단일 전송으로 제한됩니다.

        AXI Lite VALID / READY Handshake Diagram

        Figure 1-2 Fundamental Handshake

        AXI Lite의 모든 5개 채널은 동일한 VALID / READY 핸드셰이크 메커니즘을 사용하여 주소, 데이터, 응답을 전송합니다.

        • VALID 신호: 정보를 보내는 측(Source)에서 구동하며, 채널의 주소나 데이터가 유효함을 나타냅니다.
        • READY 신호: 정보를 받는 측(Destination)에서 구동하며, 데이터를 받아들일 준비가 완료되었음을 나타냅니다.
        • 전송 발생 조건: VALID와 READY가 모두 HIGH인 클럭 상승 에지에서 전송이 이루어집니다.

          AXI Lite Transaction Dependencies

          Write Path

          Write transaction dependencies

          Master-driven Slave-driven

          핵심 규칙: BVALID는 AWVALID/AWREADY와 WVALID/WREADY의 두 핸드셰이크가 모두 끝난 뒤에만 올라올 수 있습니다.

          • Master 측: AWVALID, WVALID는 각 READY를 기다리지 않고 독립적으로 구동해야 합니다.
          • Slave 측: 단순 AXI Lite slave는 AWVALID와 WVALID가 모두 준비된 시점에 AWREADY/WREADY를 함께 여는 경우가 많습니다.
          • Response: BREADY는 BVALID보다 먼저 올라와도 되고 나중에 올라와도 됩니다. 둘 다 HIGH일 때 write response가 완료됩니다.

          ARM IHI 0022H Figure A3-7 기준으로, double-headed arrow는 해당 신호가 앞선 조건이 성립한 뒤에만 assert될 수 있음을 의미합니다. 또한 BVALID는 마지막 write data beat, 즉 WLAST까지 확인된 뒤에만 생성될 수 있습니다.

          Read Path

          Read transaction dependencies

          Master-driven Slave-driven

          핵심 규칙: RVALID는 읽기 주소 핸드셰이크(ARVALID/ARREADY)가 먼저 완료되어야만 생성될 수 있습니다.

          • Master 측: ARVALID는 ARREADY를 기다리지 않고 먼저 구동해야 합니다.
          • Slave 측: ARREADY는 내부 준비 상태에 따라 지연될 수 있고, 주소를 수락한 뒤 RVALID로 데이터를 반환합니다.
          • Data Return: RREADY 역시 먼저 HIGH로 유지될 수 있으며, RVALID와 동시에 HIGH가 되는 시점에 read data transfer가 완료됩니다.

          Definition & Principle

          📚 1. 기본 정의

          이전 트랜잭션의 데이터 응답(Data Response)이 완료되기 전에 새로운 주소(Address)를 발행할 수 있는 능력을 의미합니다. '미완결(Outstanding)' 상태의 트랜잭션을 여러 개 유지할 수 있다는 뜻에서 유래되었습니다.

          ⚙️ 2. 작동 원리

          • Master는 Slave가 Busy한 상태여도 Address Channel을 통해 다음 명령을 미리 전달합니다.
          • Slave 내부의 Request Queue에 명령이 쌓이게 되며, 이는 파이프라이닝(Pipelining) 효과를 발생시킵니다.

          💡 3. 설계 이점

          Slave의 내부 처리 지연 시간(Memory Latency)을 숨길 수 있습니다. 주소를 미리 전달함으로써 데이터 채널의 유휴 시간(Idle Time)을 제거하여 대역폭 효율을 극대화합니다.

          Definition & Principle

          🔀 1. 기본 정의

          트랜잭션의 요청 순서와 상관없이, 먼저 준비된 데이터부터 먼저 응답할 수 있는 기술입니다. AXI 프로토콜의 핵심 신호인 ID Tag(ARID, RID 등)를 사용하여 각 응답이 어떤 요청에 대한 것인지 식별합니다.

          🆔 2. 작동 원리

          • Master는 각 요청에 고유한 ID 번호를 부여하여 보냅니다.
          • Slave는 처리 속도가 다른 여러 내부 자원(SRAM, DRAM 등) 중 먼저 처리가 완료된 순서대로 ID를 붙여 데이터를 반환합니다.

          🚀 3. 설계 이점

          느린 주변 장치에 대한 접근이 빠른 장치로의 접근을 방해하는 Head-of-Line Blocking 현상을 방지합니다. 시스템의 전체적인 유연성과 실질적인 데이터 처리 속도를 크게 향상시킵니다.

          Implementation Design Checklist

          ID Tagging Mechanism

          Out-of-Order를 가능하게 하는 핵심은 ID Signals입니다.

          • ● ARID: Read Address ID
          • ● AWID: Write Address ID
          • ● RID / BID: Response ID
          "마스터는 돌아온 응답의 RID를 확인하여 자신이 보낸 어떤 ARID에 대한 데이터인지 판단합니다."

          Ordering Rules

          1 Same ID Requirement

          동일한 ID를 가진 트랜잭션은 반드시 발행된 순서(In-Order)대로 완료되어야 합니다.

          2 Different ID Flexibility

          서로 다른 ID 사이에는 순서 보장 의무가 없으므로 Out-of-Order가 적극 허용됩니다.

          Hardware Overhead

          REORDER BUFFER 마스터는 응답을 정렬하기 위한 로직과 버퍼가 필요합니다.
          QUEUE DEPTH 슬레이브는 Outstanding 요청을 보관할 Queue가 필요합니다.

          * Outstanding Depth가 늘어날수록 성능은 비례하지만 하드웨어 면적(Area) 또한 비례하여 증가합니다.

          Master / Slave Structure

          RTL Architecture Guide
          Master Side: ID Management & ROB
          Mechanism 01 Transaction ID Tracking Table

          Master는 발행한 모든 Outstanding 트랜잭션을 추적하기 위한 Internal Scoreboard가 필요합니다.

          Tracking Logic
          • 1 ID Pool: 사용 가능한 ID 비트 수 2ID_WIDTH 내에서 고유 ID를 할당하여 발행합니다.
          • 2 Response Matching: RID 가 돌아오면 해당 ID의 상태를 완료로 업데이트하고 리소스를 해제합니다.
          • 3 Reorder Buffer (ROB): CPU와 같이 순서가 중요한 Master라면, OoO로 돌아온 데이터를 원래 순서대로 정렬하기 위한 대규모 Buffer를 유지해야 합니다.
          Slave Side: Multi-threaded Back-end
          Mechanism 02 Request Queueing & Banking

          Slave는 다수의 요청을 동시에 수용하고, 처리 준비가 된 장치부터 응답을 보낼 수 있는 병렬 구조를 가집니다.

          Backend Architecture
          • 1 Command FIFO: Address Phase에서 수신한 AR 정보를 저장하여 Outstanding 명령을 유지합니다.
          • 2 Memory Interleaving: 주소 공간을 여러 Bank로 나누어, 한 Bank가 작업 중일 때 다른 Bank가 응답할 수 있도록 설계합니다.
          • 3 Arbiter: 처리된 여러 응답 중 어떤 ID를 먼저 Bus로 실어 보낼지 결정하는 Response Arbiter가 필요합니다.
