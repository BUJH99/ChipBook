Solution

System Integration View

Solution은 블록을 따로 설명하는 것이 아니라, 제품처럼 묶어서 설명하는 관점입니다

SoC에서 좋은 solution은 CPU, NPU, memory subsystem, on-chip bus, off-chip interface, software bring-up까지 한 덩어리로 맞물릴 때 완성됩니다. 그래서 solution page는 “무슨 IP가 있나”보다 “어떻게 연결해 제품 문제를 푸나”에 집중하는 것이 좋습니다.

Architecture / Integration / Bring-up
Perspective
Subsystem 단위 사고

DDR controller, PHY, scheduler, cache, AXI interconnect를 따로 보지 않고 memory solution으로 봅니다.

Execution
HW + SW 연결

register map, interrupt, boot flow, driver model까지 이어져야 실제 product solution이 됩니다.

Evaluation
성능/전력/면적 trade-off

통합 관점에서는 block 개별 최적화보다, 전체 시스템 bottleneck 제거가 더 중요할 때가 많습니다.

10-2. Solution Thinking

solution 관점에서 자주 묻는 질문

이 제품의 병목은 어디인가?

compute가 느린지, memory bandwidth가 모자란지, bus contention이 심한지 먼저 분리해서 봅니다.

HW와 SW 경계는 어디서 맞물리나?

register map, interrupt, DMA descriptor, driver API가 실제 제품 완성도를 좌우합니다.

확장성이 있는가?

1개의 accelerator에서 4개로 커졌을 때 bus와 memory 구조가 버틸 수 있는지까지 봐야 합니다.
