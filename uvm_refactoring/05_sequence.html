UVM Guidebook Sequence

1 Sequence의 역할

Sequence는 DUT(Device Under Test)에 인가할 자극(Stimulus)인 Transaction(Item)을 생성하고, 동작의 시나리오를 구성하는 핵심 객체입니다.

Sequence는 생성된 Item을 Driver에게 직접 전달하지 않습니다. 대신, Sequencer라는 중개자를 거쳐 Driver가 요청할 때마다 데이터를 공급하는 구조를 가집니다. 이를 통해 복잡한 시나리오를 모듈화하고 재사용성을 극대화할 수 있습니다.

Data Flow Scenario

2 uvm_sequence 상속 구조

Sequence는 테스트 벤치의 계층 구조에 고정된 Component(예: Env, Agent)가 아닙니다. 수명이 짧고 동적으로 생성 및 소멸되는 Object(객체)입니다. uvm_object를 최상위로 하여 상속받으며, 동적 생성 및 Factory 오버라이딩 기능을 완벽하게 지원합니다.

Inheritance Tree

my_sequence.sv 
class my_sequence extends uvm_sequence #(my_item);

  // 5. Factory Registration
  `uvm_object_utils(my_sequence)

  // 4. Constructor
  function new(string name = "my_sequence");
    super.new(name);
  endfunction

  // 6. task body() 
  virtual task body();
    // ... item creation and stimulus ...
  endtask

endclass

3 Parameterized Type : #(ItemType)

uvm_sequence #(my_item)와 같이 클래스를 정의할 때 파라미터를 통해 Sequence가 다룰 Sequence Item의 타입을 명시합니다. 이를 통해 컴파일러는 타입 안정성을 보장하며, 내부의 req와 rsp 핸들이 해당 타입으로 자동 선언됩니다.

4 Constructor : new() / super.new()

Sequence는 컴포넌트 계층 트리에 속하지 않으므로, new() 함수의 인자로는 오직 name만 필요합니다. (parent 인자가 없음). 내부에서 반드시 super.new(name)을 호출하여 상위 클래스를 초기화해야 합니다.

5 Factory Registration

`uvm_object_utils(my_sequence) 매크로를 사용하여 Factory에 등록합니다. Sequence는 Component가 아니기 때문에 _component_utils가 아닌 _object_utils를 사용합니다. 이를 통해 테스트 단위에서 다른 Sequence로의 오버라이딩(Overriding)이 가능해집니다.

6 task body() 와 Item 생성 라이프사이클

body() 태스크는 Sequence의 심장부입니다. Sequence가 start() 메소드에 의해 구동되면 자동으로 호출되며, 이곳에 Transaction(Item)을 생성하고 Driver로 보내는 시나리오가 작성됩니다. 다음 다이어그램은 body() 내부에서 단일 Item이 처리되는 표준 흐름을 보여줍니다.

7 Item Creation

req = my_item::type_id::create("req");
Factory 메커니즘을 통해 Transaction 객체를 생성합니다. 일반적인 new() 대신 create()를 사용하여 향후 다형성을 통한 Test 계층에서의 객체 오버라이딩을 지원합니다.

8 start_item(item)

생성된 객체를 Driver로 보내기 위한 첫 단계입니다. 이 함수는 비동기적으로 동작하며, Sequencer에게 버스 사용 권한(Grant)을 요청하고 승인이 떨어질 때까지 대기(Block)합니다.

9 item.randomize()

Sequencer로부터 승인을 얻은 직후, 실제로 데이터를 전송하기 전(finish_item 직전)에 데이터를 Randomize 합니다. 이를 Late Randomization이라고 하며, 시뮬레이션의 가장 최신 상태를 반영한 무작위 값을 생성할 수 있어 매우 중요합니다.

10 Local / Inline Constraint

if (!req.randomize() with { addr == 32'hA000; })

기본 Item 클래스에 정의된 Constraint 외에, 특정 Sequence 시나리오에서만 유효한 제약 조건을 with 구문을 사용하여 즉석에서(Inline) 주입합니다. 이를 통해 Item 클래스의 수정 없이 유연한 시나리오 구성이 가능합니다.

Item Lifecycle 설계 의도 (Intent Analysis)

UVM Sequence에서 Item이 생성되고 전달되는 각 단계는 단순한 함수 호출이 아닌 독립적인 설계 의도(Intent)를 가지고 있습니다. 다음 다이어그램은 각 단계별 핵심 목적과 아키텍처적 이점을 시각화합니다.

item() 라이브러리 함수 내부: Sequence-Driver TLM Handshake

start_item()과 finish_item() 함수 내부에서는 사실 Sequencer를 중개자로 하여 Driver와 타이밍을 맞추는 복잡한 양방향 통신(Handshake)이 일어납니다. 아래 시퀀스 다이어그램은 시간에 따른 각 컴포넌트의 내부 호출 상태를 시각화한 것입니다.

1. Driver's Request

Driver는 구동할 데이터가 필요할 때 get_next_item()을 호출합니다. 만약 Sequencer에 준비된 Item이 없다면, 이 함수는 Blocking 상태로 대기합니다.

2~5. Sequence Handshake

Sequence가 start_item()을 호출하면 Sequencer가 중재(Arbitration)를 수행합니다. Driver가 대기 중이므로 즉시 Grant를 주고, 값을 Randomize한 후 finish_item()을 통해 Item을 보냅니다.

6~8. Driver Completion

데이터를 받은 Driver는 물리적 핀(Pin)에 신호를 인가합니다. 구동이 끝나면 item_done()을 호출하여 완료를 알리고, 이때 Blocking되어 있던 finish_item()이 해제되며 단일 사이클이 종료됩니다.

UVM 내부 실제 흐름 및 소스 코드 로직

단순해 보이는 start_item()과 finish_item() 태스크는 UVM Base Class 내부적으로 검증, 권한 획득, 그리고 다양한 Callback(pre_do, mid_do, post_do)을 호출하는 견고한 로직을 포함하고 있습니다.

uvm_sequence_base.svh 
task start_item(uvm_sequence_item item);
  // 1. item이 null이면 fatal 에러 발생
  if (item == null) `uvm_fatal("SEQ", "Item is null");

  // 2. sequence가 아니라 진짜 item인지 검사
  if (!item.is_item()) `uvm_fatal("SEQ", "Not an item");

  // 3. sequencer 찾기
  sequencer = item.get_sequencer();
  if (sequencer == null)
      sequencer = get_sequencer();

  // 4. item에 parent sequence와 sequencer 연결
  item.set_item_context(this, sequencer);

  // 5. priority(우선순위) 결정
  if (priority < 0)
      priority = get_priority();

  // 6. sequencer grant 대기 (Driver가 받을 준비가 될 때까지 Blocking)
  sequencer.wait_for_grant(this, priority);

  // 7. pre_do callback 호출 (Randomize 직전에 개입 가능)
  pre_do(1);
endtask
task finish_item(uvm_sequence_item item);
  // 1. item에 연결된 sequencer 가져오기
  sequencer = item.get_sequencer();
  if (sequencer == null)
      `uvm_fatal("SEQ", "Sequencer is null");

  // 2. 보내기 직전 callback 호출
  mid_do(item);

  // 3. sequencer로 request(실제 데이터) 전송
  sequencer.send_request(this, item);

  // 4. driver가 item_done() 할 때까지 대기 (Blocking)
  sequencer.wait_for_item_done(this, -1);

  // 5. 완료 후 callback 호출
  post_do(item);
endtask

Callback (pre_do, mid_do, post_do)의 중요성: 사용자는 기본 Sequence 로직을 수정하지 않고도, 상속받은 Sequence에서 해당 함수들을 override하여 특정 시점에 원하는 로직(예: 디버깅 출력, 강제 제약조건 삽입, 완료 후 스코어보드 알림 등)을 손쉽게 끼워 넣을 수 있습니다.
