1. Verification Architecture

[1] VCS / Verdi 개념

VCS는 고성능 컴파일 기반의 시뮬레이터로, Verilog/SV 코드를 C/C++ 기반의 실행 파일(simv)로 변환합니다. Verdi는 생성된 데이터베이스(.fsdb, kdb)를 로드하여 강력한 시각적 디버깅 환경을 제공합니다.

[2] Simulation 기본 흐름

전체 검증 사이클은 Code 작성 ➔ File List 구성 ➔ VCS Compile (분석/빌드) ➔ 시뮬레이션 실행 (파형 Dump) ➔ Verdi 분석 ➔ 수정의 순환 고리로 이루어집니다. 이 과정을 자동화하는 것이 핵심입니다.

[11] 연동 구조 (KDB)

과거에는 Verdi용 vericom 컴파일을 따로 수행했지만, 최근 Flow에서는 VCS 컴파일 시 -kdb (Knowledge Database) 옵션을 주입하여 소스 및 스키마틱 정보를 Verdi로 단번에 전달(Unified Flow)합니다.

      -kdb -lca -debug_access+all

[3] VCS 내부 컴파일 엔진 구조 (Under the Hood)

VCS(Verilog Compiler Simulator)는 단순한 코드 해석기(Interpreter)가 아닙니다. 하드웨어 코드를 C/C++로 변환하고 GCC 컴파일러를 통해 네이티브 바이너리 머신 코드(simv)로 빌드하는 고성능 컴파일 엔진입니다.

1
Analysis (구문 분석)

vlogan 명령어가 Verilog/SV 파일을 읽어 문법적 오류를 검사하고, 내부적으로 추상 구문 트리(AST) 형태의 중간 데이터베이스를 생성합니다.

2
Elaboration (구조화 및 C 변환)

vcs 엘레보레이터가 모듈들의 계층(Hierarchy) 구조를 연결하고 Parameter를 계산합니다. 가장 중요한 점은 이 단계에서 시뮬레이션을 위한 C/C++ 소스코드(vcs_main.c 등)가 동적으로 자동 생성된다는 것입니다.

3
Compilation (바이너리 빌드)

생성된 C/C++ 코드를 시스템의 GCC나 Clang 컴파일러가 컴파일하고, 시뮬레이션 커널 라이브러리와 링킹(Link)하여 호스트 머신에서 직접 실행 가능한 초고속 바이너리 파일인 simv를 출력합니다.

VCS Essential Command & Option Dictionary

-full64
64비트 모드로 컴파일 및 시뮬레이션을 수행합니다. 대규모 디자인 검증 시 메모리 부족(OOM)을 방지하기 위해 실무에서는 필수적으로 사용합니다.
-j<N> (e.g., -j8)
C/C++ 코드 컴파일 시 사용할 멀티 코어(스레드) 개수를 지정하여 빌드 속도를 대폭 단축시킵니다.
-timescale=1ns/1ps
시뮬레이션의 기준 시간 단위(Time unit)와 정밀도(Precision)를 전역으로 설정합니다. 모듈 간 timescale 충돌을 방지합니다.
+incdir+<path>
Verilog 헤더 파일(`include "macro.vh")을 찾을 디렉토리 경로를 추가합니다. 여러 개 지정 가능합니다.
+define+<MACRO>
소스코드 내의 `ifdef MACRO 구문을 활성화하기 위해 컴파일 타임에 매크로 변수를 정의합니다.
-parameters <file>
Top 모듈의 Parameter 값을 소스코드 수정 없이 외부 파일에서 강제로 덮어씌웁니다(Override).

3. Simulation Execution

[6] Simulation 실행  |  [7] Waveform Dump 구조

[7] FSDB Dump 최적화

VCD 대비 압축률이 높아 대규모 검증에 필수적입니다. Testbench의 initial 블록 내에서 시스템 태스크를 통해 영역을 세밀하게 지정합니다.

$fsdbDumpfile("test.fsdb");
$fsdbDumpvars(0, tb_top); // 0: All hierarchy
$fsdbDumpSVA; // Assertion Dump

[6] 런타임 옵션 (Plusargs)

재컴파일 없이 시뮬레이션 동작을 제어하기 위해 + 기호로 시작하는 인자를 전달합니다.

  • +UVM_TESTNAME=test_01 실행할 UVM 테스트 지정
  • +ntb_random_seed=123 난수 시드 고정 (재현성 확보)

[8] 기본 창 (Windows)

  • nWave: 신호 파형 확인 창
  • nTrace: RTL 소스코드 탐색 및 신호 역추적(Trace)
  • nSchema: RTL 코드를 Gate-level 회로도로 시각화

[9] Trace & Driver

파형에서 알 수 없는 값(X)이나 Error 발생 시, 해당 신호를 더블클릭하여 값을 생성한 원인 소스코드로 즉각 점프합니다.

SVA [12] Assertion Debug

SVA 실패 지점을 파형에 마커로 표시하며, Active Window 창을 통해 시퀀스 조건을 시각적으로 쪼개어 분석합니다.

[18] 실전 Debug 시나리오

  • 파형에서 X 상태 식별
  • Active Trace로 Source 역추적
  • FSM View로 State 락(Lock) 원인 파악

[11] Verdi 통합 디버깅 아키텍처 (Under the Hood)

Verdi는 단순한 파형 뷰어(Waveform Viewer)가 아닙니다. 동적 시뮬레이션 결과(FSDB)와 정적 설계 정보(KDB)를 하나의 강력한 엔진으로 결합하여, 소스코드-파형-회로도 간의 완벽한 양방향 추적(Cross-probing)을 지원하는 통합 플랫폼입니다.

1
FSDB (동적 데이터)

시뮬레이션 실행 중 시간에 따른 신호의 변화값(Value Change)을 저장한 고압축 파일입니다. nWave에 타이밍과 파형을 그리는 핵심 소스입니다.

2
KDB (정적 설계 정보)

VCS 컴파일 시 -kdb 옵션으로 생성되는 Knowledge DB입니다. 계층 구조, 소스 코드 라인, 파라미터 값 등을 담아 simv.daidir에 저장되며 nTrace와 nSchema를 구성합니다.

3
Cross-Probing (상호 연동)

두 DB가 결합되어, 파형(nWave)에서 특정 시간을 클릭하면 해당 시점의 코드가 nTrace에 강조(Active Annotation)되고, 회로도(nSchema) 상의 드라이버를 즉시 추적할 수 있습니다.

>_ Verdi Execution

RTL + Waveform 동시 로드
verdi -f filelist.f -ssf test.fsdb
KDB (컴파일된 DB) 바로 열기
verdi -dbdir simv.daidir -ssf test.fsdb
시작 로고 무시 및 저장된 신호(TCL) 복원
verdi -nologo -play signal.tcl

5. Advanced & Automation

Regression & Coverage Merge Flow

[10] UVM + VCS Flow

VCS는 UVM 라이브러리를 내장하고 있어 -ntb_opts uvm 옵션 하나로 복잡한 검증 환경을 즉시 빌드할 수 있습니다.

[15] Regression & [16] Log 파싱

서버 팜(LSF 등)에서 수십 개의 테스트케이스를 병렬로 돌린 후, Python/Perl 스크립트를 통해 시뮬레이션 로그 파일(sim.log) 내의 "UVM_ERROR" 또는 "TEST PASSED" 문자열을 파싱하여 결과를 자동 집계합니다.

[13] Coverage 병합 (URG)

개별 실행으로 생성된 분산 Coverage DB(.vdb)들을 Synopsys URG 유틸리티로 병합하여 최종 검증 커버리지 리포트(HTML)를 생성합니다.

[17] 자주 발생하는 Error

  • Error-[SFCOR]: 소스코드 내 Syntax 오류.
  • Error-[URMI]: Module Instantiation 불가 (File list 경로 누락).
  • License Error: 환경변수 LM_LICENSE_FILE 확인 요망.

6. Ecosystem & SW Integration

[22] Git & CI/CD  |  [23] LSF Server Farm  |  [24] Scripting (Python/TCL)

실무 파이프라인: 타 SW(버전관리, 스케줄러, 스크립트)와의 완벽한 연동

현업에서는 개발자가 직접 시뮬레이션을 돌리기보다, 자동화된 파이프라인과 커스텀 스크립트를 통해 Tool의 한계를 극복합니다.

1
CI/CD & Job Scheduler

수만 줄의 검증 코드는 개인 PC에서 돌릴 수 없습니다. 코드를 Git에 Push하면 Jenkins가 트리거되어 LSF(bsub)나 SGE를 통해 수십~수백 대의 서버로 VCS 시뮬레이션을 분산 할당(Job Dispatch)합니다.

bsub -q normal -J "sim_test" vcs ...
2
Python Log Parsing

수천 개의 시뮬레이션 로그(sim.log)를 사람이 일일이 읽지 않습니다. Python이나 Perl 정규식(RegEx) 스크립트를 파이프라인에 엮어 UVM_ERROR나 FAIL 키워드만 추출해 Jira 티켓을 자동 생성하거나 Slack으로 알림을 보냅니다.

3
Verdi TCL Automation

Verdi의 모든 GUI 액션은 내부적으로 TCL 커맨드로 동작합니다. 매번 신호를 드래그하지 않고, 중요 신호 그룹, 색상, 아날로그 파형 변환 셋업을 *.tcl로 저장하여 실행 시 옵션으로 붙여 시간을 극단적으로 절약합니다.

verdi -play auto_signal_load.tcl

[25] 실무 확장: Formal & Emulation Tool 연동

VCS/Verdi가 생성한 KDB와 컴파일 데이터는 Synopsys 생태계 내 다른 툴과 완벽히 융합됩니다. 복잡한 코너 케이스 증명을 위해 JasperGold (Formal Verification)로 KDB를 넘기거나, 시뮬레이션 속도 한계를 극복하기 위해 ZeBu / Palladium (Hardware Emulator) 장비로 매핑할 때, VCS 빌드 환경(File list, Macro, Option)이 그대로 재사용됩니다.
