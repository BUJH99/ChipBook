DIGITAL DESIGN GUIDE

2. SV/OOP/UVM 기초

SystemVerilog의 객체 지향 프로그래밍(OOP) 개념과 검증 환경 구축을 위한 핵심 기반 요소들을 시각적으로 이해합니다.

1 Function vs Task

function
시뮬레이션 시간을 소모하지 않음. 하드웨어의 Combinational logic이나 단순 계산에 사용.
task
#delay, @event 등 시간 소모 구문 포함 가능. 시퀀스 구동에 필수적.

2 class vs module

Architecture Concept
Memory Allocation
module Static / Hardware
  • 용도: 실제 칩으로 합성(Synthesis)될 하드웨어 구성요소(RTL)를 모델링합니다.
  • 메모리 할당: 컴파일 시점(Compile Time)에 Static Data 영역에 고정 할당되며, 시뮬레이션 시작부터 종료 시까지 메모리에서 사라지지 않습니다.
  • 특징: always, initial 블록을 가지며, wire나 reg를 통해 물리적인 포트 연결(Hierarchical connection)을 가집니다. 상속이나 다형성을 지원하지 않습니다.
class Dynamic / OOP
  • 용도: 유연하고 재사용 가능한 검증 환경(TB, UVM 컴포넌트, 트랜잭션)을 구축하는 데 사용됩니다.
  • 메모리 할당: 실행 시점(Run Time)에 new() 함수를 호출하여 Heap 영역에 동적으로 객체(Object)를 생성합니다.
  • 특징: 아무도 참조하지 않는 객체는 Garbage Collector에 의해 자동으로 메모리에서 해제됩니다. 상속(Inheritance)과 다형성(Polymorphism)을 완벽히 지원합니다.

3 new, super, this

객체의 라이프사이클(생성)과 클래스 계층 간의 멤버 접근 및 자기 참조를 관리하는 필수 키워드

1. Constructor: new()
2. Parent Access: super
3. Self Reference: this
new()

생성자 (Constructor)

클래스의 인스턴스(객체)를 동적으로 생성합니다. 호출 시 Heap 영역에 메모리를 할당하고 모든 내부 변수를 기본값으로 초기화한 뒤, 해당 메모리 주소(Handle)를 반환합니다. 파라미터를 넘겨 초기화할 수도 있습니다.

super

부모 참조 (Parent Scope)

자식 클래스 내부에서 부모 클래스의 멤버(변수, 메서드)에 접근할 때 사용합니다. 특히 자식이 부모의 메서드를 오버라이드(Override)했을 때, 부모의 원래 기능(super.build())을 먼저 실행하기 위해 필수적으로 사용됩니다.

this

자기 참조 (Self Reference)

현재 실행 중인 객체 자신의 핸들을 의미합니다. 메서드의 인자 이름과 클래스의 멤버 변수 이름이 동일할 때, 스코프의 모호성을 해결하고 명확히 클래스 멤버 변수를 지칭(this.data)하기 위해 주로 사용됩니다.

4, 5 virtual, inheritance, polymorphism

객체 지향 프로그래밍(OOP)의 핵심: 클래스 확장과 동적 바인딩을 통한 유연한 검증 환경 구축

1. Inheritance (상속)
2. Polymorphism & Virtual Binding
extends

Inheritance (상속)

기존 클래스(Base)의 변수와 메서드를 물려받아 새로운 클래스(Child)를 생성합니다. 중복 코드를 줄이고 검증 컴포넌트의 재사용성을 극대화합니다. 자식 클래스는 부모의 메서드를 재정의(Override)할 수 있습니다.

Base h = new Child

Polymorphism (다형성)

부모 타입의 핸들로 자식 타입의 객체를 가리킬 수 있는 성질입니다. 이 덕분에 검증 환경(Env) 코드를 수정하지 않고도, 외부에서 다양한 형태의 트랜잭션이나 컴포넌트(자식 객체)를 주입하여 동작시킬 수 있습니다.

virtual

Dynamic Binding (동적 바인딩)

메서드 앞에 virtual을 붙이면 컴파일러는 핸들의 타입이 아닌, "실제로 메모리에 생성된 객체"를 런타임에 확인하여(vtable 참조) 해당 객체의 오버라이드된 메서드를 실행합니다. 다형성 구현의 핵심입니다.

6 Parameterized Class

하나의 코드로 여러 데이터 타입을 지원하는 클래스 템플릿 (Type Safety & 재사용성 극대화)

Class Template Factory
#(type T)

개념 및 장점 (Concept)

  • 코드 재사용성: `int`용 mailbox, `Packet`용 mailbox를 따로 만들 필요 없이, 하나의 템플릿(Blueprint) 코드를 작성하여 다양한 타입에 재사용합니다.
  • Type Safety (타입 안정성): `mailbox #(Packet)`으로 생성된 객체에 `int` 값을 넣으려 시도하면 컴파일 에러가 발생하여, 런타임 버그를 미연에 방지합니다.
  • 값(Value)도 파라미터로 넘길 수 있습니다. 예: class Array #(int SIZE=10);
UVM Usage

UVM에서의 활용

UVM(Universal Verification Methodology) 프레임워크의 거의 모든 데이터 통신 및 제어 컴포넌트가 이 Parameterized Class로 구현되어 있습니다.

uvm_analysis_port #(my_transaction) ap;

class my_driver extends uvm_driver #(my_transaction);

7 package / import / include

  • package/import: 네임스페이스를 생성하여 이름 충돌을 방지하고 깔끔하게 컴포넌트를 가져옴 (권장).
  • `include: C언어처럼 파일의 텍스트를 물리적으로 복사해 붙여넣음. (매크로 정의 등에 주로 사용)

8 Makefile / Compile Flow

반복적인 컴파일과 시뮬레이션 과정을 자동화하는 Makefile 의존성 구조와 실전 UVM 스크립트 작성법

Makefile Dependency & Data Flow
Makefile (UVM Template) 
# --- Variables (설정 변수) ---
TEST_NAME  ?= base_test
SEED       ?= random

# 컴파일 옵션: SV 활성화, UVM 라이브러리 연동, 디버그 모드
COMP_OPTS   = -sverilog -ntb_opts uvm -timescale=1ns/1ps \
              -debug_access+all -l comp.log
# 시뮬레이션 옵션: UVM 테스트 지정, GUI 옵션 연동
SIM_OPTS    = +UVM_TESTNAME=$(TEST_NAME) -ntb_random_seed=$(SEED) \
              -l sim.log

# --- Targets (실행 타겟) ---
# make all 실행 시 compile 후 sim 실행 (의존성)
all: compile sim

compile:
	vcs $(COMP_OPTS) -f flist.f

sim:
	./simv $(SIM_OPTS)

# Verdi (파형 뷰어) 띄우기
wave:
	verdi -ssf novas.fsdb &

clean:
	rm -rf csrc simv simv.daidir *.log *.fsdb

1. 매크로/변수 (Variables)

TEST_NAME처럼 자주 바뀌는 값을 변수로 설정합니다. 터미널에서 make sim TEST_NAME=error_test 형태로 외부에서 인자를 주입하여 스크립트 수정 없이 다양한 테스트를 구동할 수 있습니다.

2. 컴파일 옵션 (vcs flags)

  • -ntb_opts uvm: 툴(VCS)에 내장된 UVM 라이브러리를 자동으로 맵핑합니다.
  • -debug_access+all: Verdi 파형(.fsdb)을 뽑기 위한 디버깅 권한을 활성화합니다.
  • -f flist.f: 수십 개의 RTL/TB 파일 경로를 모아둔 텍스트 파일을 읽어옵니다.

3. 시뮬레이션 옵션 (simv flags)

+UVM_TESTNAME은 UVM 내부의 run_test() 함수로 전달되어 지정한 이름의 테스트 클래스를 동적으로 생성하여 검증을 시작하는 가장 중요한 인자입니다.
