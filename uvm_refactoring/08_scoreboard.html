Scoreboard : Compare DUT Result with Golden Model

Scoreboard는 DUT(Design Under Test)의 실제 출력 결과와 Golden Model(Reference Model)이 생성한 기대값(Expected Value)을 비교하여 검증의 성공 여부를 판별하는 핵심 컴포넌트입니다. 단순히 값을 비교하는 것을 넘어, 데이터의 누락(Drop), 순서 뒤바뀜(Out-of-order), 중복(Duplication) 등의 치명적인 결함을 잡아내는 UVM 검증 환경의 최종 판사 역할을 수행합니다.

구조 및 연결 (Architecture & Connection)

uvm_scoreboard 상속

UVM의 컴포넌트이며 uvm_scoreboard를 상속받습니다. 검증 환경의 최상위(Env) 레벨에서 인스턴스화됩니다. 내부적으로 하나 이상의 비교 로직(Evaluator)을 포함할 수 있습니다.

Port to Imp 연결

Monitor의 uvm_analysis_port에서 브로드캐스팅된 트랜잭션은 Scoreboard의 uvm_analysis_imp로 전달됩니다. 이는 TLM(Transaction Level Modeling) 기반의 비동기적 통신 방식으로, Monitor는 수신자를 몰라도 데이터를 던질 수 있어 결합도(Coupling)를 극도로 낮춥니다.

write() Function

Monitor가 port.write()를 호출하면, 연결된 Scoreboard에 구현된 write() 함수가 0-time(즉시)에 자동으로 실행되어 데이터를 수신하고 이를 큐(Queue)에 저장하거나 즉각적인 비교를 수행합니다.

Golden Model & 비교 로직

[6] Reference Model  |  [7] Expected vs Actual

Golden Model은 완벽하게 동작하는 소프트웨어 레퍼런스 모델입니다. 주로 C/C++(DPI-C 연동) 또는 High-level SystemVerilog로 작성되며, 하드웨어의 클럭(Cycle) 주기와 무관하게 기능적 정확성(Functional Accuracy)에 초점을 맞추어 Expected Data(기대값)를 생성합니다.

Scoreboard의 Comparator 로직은 이 기대값을 DUT의 출력인 Actual Data(실제값)와 비교합니다. 이때 단순한 Payload(데이터 내용) 비교 외에도 프로토콜에 따른 헤더 정보, 에러 플래그 등 트랜잭션 객체 내의 모든 필수 필드를 검증하게 됩니다.

비교 순서 정책 (Ordering Policies)

[11] In-order vs Out-of-order Compare 및 데이터 구조 설계

In-order 순차 비교

적용 인터페이스: FIFO, 단순 파이프라인(Pipeline) 연산기, UART, SPI 등 데이터가 입력된 순서대로 반드시 출력되는 시스템.

UVM 구현 핵심 (Queue): SystemVerilog의 일반 큐(Queue) 구조인 tx exp_q[$]를 사용합니다. Golden Model이 데이터를 생성하면 push_back()으로 큐의 꼬리에 넣고, Monitor에서 실제값이 도착하면 pop_front()로 큐의 머리에서 데이터를 꺼내어 1:1로 즉시 비교합니다.

⚠️ 주의점: 중간에 데이터가 하나라도 누락(Drop)되거나 순서가 어긋나면, 큐의 인덱스가 엇갈리게 되어 그 이후의 모든 비교가 연속해서 Mismatch를 발생시키는 'Cascade Error'가 일어날 수 있습니다.

Out-of-order 비순차 비교

적용 인터페이스: PCIe, AXI 버스(Out-of-order completion 지원), 멀티 포트 네트워크 라우터 등 우선순위(QoS)에 의해 패킷 처리 순서가 뒤바뀔 수 있는 시스템.

UVM 구현 핵심 (Associative Array): 트랜잭션의 고유 식별자(Transaction ID, Address 등)를 Key로 하는 연관 배열(Associative Array) tx exp_aa[int]를 사용합니다. 실제값이 도착하면 exp_aa.exists(act_t.id)로 배열 전체를 탐색(Lookup)하여 해당 ID의 기대값을 꺼내 비교합니다.

💡 가비지 컬렉션(GC): 비교가 성공적으로 완료된 트랜잭션은 메모리 누수(Memory Leak)를 막기 위해 반드시 exp_aa.delete(act_t.id)를 호출하여 배열 공간을 비워주어야 합니다.

Scoreboard 구현 아키텍처 유형

[8] 즉시 비교  |  [9] FIFO 기반  |  [10] 다중 모니터 기반 처리 구조

Immediate 즉시 비교형 (Combinational)

타이밍 지연(Latency)이 없는 순수 조합 논리회로(Combinational Logic)나, 입력과 동시에 결과가 도출되는 시스템을 검증할 때 사용합니다.

동작 원리: 데이터를 메모리(Queue, AA)에 저장하지 않습니다. write() 함수가 호출되는 즉시 양쪽 포트의 데이터를 1:1로 비교합니다. 두 데이터가 정확히 같은 시뮬레이션 타임(0-delay)에 도착한다는 보장이 있어야 합니다.

TLM FIFO 지연 보상형 (Latency-Aware)

파이프라인이나 버퍼링이 존재하여 입력(Stimulus)과 출력(Response) 사이에 수~수백 사이클의 지연이 발생하는 시스템을 위한 가장 표준적인 UVM 모델입니다.

동작 원리: Golden Model이 생성한 Expected 데이터를 uvm_tlm_analysis_fifo에 버퍼링합니다. 이후 DUT에서 연산이 완료되어 Actual 데이터가 도착하면(Event 발생), FIFO에서 get() 함수를 호출하여 Expected 데이터를 꺼내 비교합니다.

Multi-Port 다중 모니터 종합 처리

SoC 레벨 검증이나 멀티 포트 IP (예: AXI Interconnect, Network Switch, Multi-channel DMA) 구조에서 사용됩니다.

동작 원리: 단순히 1:1 비교를 넘어서, 여러 입력 포트의 트랜잭션을 내부 로직(Associative Array 등)에서 조립(Assembly)하거나 패킷 라우팅 헤더를 확인하여 예상되는 출력 포트의 Actual 데이터와 교차 검증(Cross-check)을 수행합니다.

※ UVM Macro `uvm_analysis_imp_decl 사용 필수 구역

검증 결과 정책 및 리포트

[12] Error/Warning 정책  |  [13] report_phase

UVM_INFO @ 100ns: reporter [SB_MATCH] Exp: 0x4A, Act: 0x4A
UVM_INFO @ 120ns: reporter [SB_MATCH] Exp: 0xFF, Act: 0xFF
UVM_ERROR @ 150ns: reporter [SB_MISMATCH] Exp: 0x12, Act: 0x10
UVM_WARNING @ 180ns: reporter [SB_QUEUE] Expected queue is getting full (100 items)
Passed (Match) 1,204
Failed (Mismatch) 1

check_phase: 시뮬레이션의 메인 동작이 끝난 직후 실행됩니다. Scoreboard 내부의 Expected Queue나 Associative Array에 비교되지 못하고 남아있는 데이터(Orphan Transaction)가 없는지 최종 검사합니다. 데이터가 남아있다면 DUT 내부에서 패킷이 소실(Drop)되었거나 데드락(Hang)이 발생했음을 강력하게 시사하는 치명적 오류입니다.

report_phase: 검증의 가장 마지막 단계입니다. 총 매칭(Match) 횟수, 불일치(Mismatch) 횟수, 발생한 Error/Warning의 통계를 종합하여 리포팅합니다. UVM 정책상 UVM_ERROR가 단 한 번이라도 발생하면 해당 시뮬레이션 테스트는 최종 FAILED 상태로 종료됩니다.

실제 코드 구조

[14] SystemVerilog UVM Code Skeleton

my_scoreboard.sv 
class my_scoreboard extends uvm_scoreboard;
  `uvm_component_utils(my_scoreboard)
  
  // [4] Analysis Implementation Ports (TLM 매크로 선언 필요)
  uvm_analysis_imp_expected #(tx, my_scoreboard) exp_imp;
  uvm_analysis_imp_actual   #(tx, my_scoreboard) act_imp;

  // [9] Queue for Expected Data
  tx exp_queue[$];
  
  function new(string name, uvm_component parent);
    super.new(name, parent);
    exp_imp = new("exp_imp", this);
    act_imp = new("act_imp", this);
  endfunction

  // [5] write() from Golden Model
  virtual function void write_expected(tx t);
    exp_queue.push_back(t);
  endfunction

  // [5] write() from DUT Monitor & [7] Compare Logic
  virtual function void write_actual(tx act_t);
    tx exp_t;
    if (exp_queue.size() == 0) begin
      `uvm_error("SB", "No expected data available!")
      return;
    end
    
    exp_t = exp_queue.pop_front(); // [11] In-order pop
    
    if (act_t.compare(exp_t)) begin
      `uvm_info("SB", "Match!", UVM_LOW)
    end else begin
      `uvm_error("SB", "Mismatch Detected!") // [12] Error Policy
    end
  endfunction
endclass
UVM Macro Tip: 다중 Imp Port 선언
기본적으로 UVM의 uvm_analysis_imp는 연결된 컴포넌트 내에 write()라는 이름의 함수가 1개 존재할 것을 요구합니다. 위 코드처럼 expected와 actual 두 개의 포트가 필요할 경우 이름 충돌이 발생합니다. 이를 회피하기 위해 `uvm_analysis_imp_decl(_expected) 및 `uvm_analysis_imp_decl(_actual) 매크로를 사전에 선언하면 write_expected(), write_actual()처럼 함수 이름을 우아하게 분리하여 구현할 수 있습니다.
