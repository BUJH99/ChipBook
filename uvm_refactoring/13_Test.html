Semiconductor Digital Circuit Design Guidebook

UVM Test : Control Verification Scenario

UVM 1.2 SystemVerilog

1. 구조 및 역할 (Hierarchy & Role)

Test는 UVM 검증 환경의 최상위 컴포넌트(uvm_test_top)입니다. 모든 시나리오는 공통 환경(Base Test)을 상속받아 코드 중복을 최소화하고, 다형성(Polymorphism)을 활용해 시나리오별 파생 Test를 효율적으로 관리합니다.

[1] Test의 역할 & [14] Phase별 역할

  • ▸
    build_phase: 하위 컴포넌트인 Env(Environment)를 생성하고, 전체 검증 환경의 토폴로지를 조립합니다. Top-down 방식으로 실행됩니다.
  • ▸
    run_phase: Time-consuming phase로, 실제 시뮬레이션 시간을 소모하며 트랜잭션을 발생시키는 시나리오(Sequence)를 구동합니다.

[2] 상속구조 & [12] Scenario 분리

  • ▸
    uvm_test 상속: 모든 테스트는 기본 UVM 베이스 클래스를 상속받아야 합니다.
  • ▸
    Base Test 분리: Timeout 설정, 공통 Sequence 실행, Env 생성 등 모든 시나리오에 공통되는 요소는 base_test에 구현합니다.
  • ▸
    Scenario Test 분리: 각 테스트 시나리오(예: Error Injection)는 base_test를 다시 상속받아 필요한 부분만 재정의하여 사용합니다.

2. 환경 구성 (build_phase & Config)

Test의 build_phase에서는 Factory 패턴을 활용해 동적으로 Env를 생성합니다. 또한 Config DB를 통해 하위 컴포넌트에 필요한 가상 인터페이스(Virtual IF)나 설정 객체(Config Object)를 배포하며, 기존 코드를 수정하지 않고도 동작을 변경할 수 있는 Factory Override를 적용합니다.

[3] Env 생성) 전달된 Config와 Override Type을 적용받음.

[4] build_phase & [5] Config 설정

  • ▸
    컴포넌 생성: new() 대신 type_id::create()를 사용하여 UVM Factory를 통해 Env 및 하위 컴포넌트를 생성합니다. 이는 추후 Override를 가능하게 합니다.
  • ▸
    Config DB (uvm_config_db): TB Top에서 전달된 Virtual Interface나 구조적 설정값(예: Agent의 Active/Passive 모드)을 Env 및 하위 Agent로 전달하는 전역 데이터베이스 역할을 합니다.

[6] Factory Override

  • ▸
    개념 및 장점: 원본 클래스의 소스 코드를 단 한 줄도 수정하지 않고, 시뮬레이션 런타임에 특정 클래스를 다른 클래스(파생 클래스)로 교체하는 강력한 기능입니다.
  • ▸
    활용 사례: 정상적인 트랜잭션을 발생시키는 A_seq를 에러를 주입하는 Error_A_seq로 교체하여 손쉽게 네거티브 테스트(Negative Test)를 구성할 수 있습니다.

3. 시나리오 실행 (run_phase & Sequence)

run_phase는 시뮬레이션의 실질적인 동작이 이루어지는 단계입니다. Objection 메커니즘을 통해 시뮬레이션의 시작과 끝을 제어하며, SoC/System 레벨에서는 Virtual Sequence를 사용하여 여러 하위 에이전트들의 트랜잭션 흐름을 오케스트레이션(Orchestration) 합니다.

[8] Objection Mechanism

  • ▸
    백그라운드 프로세스가 완료될 때까지 시뮬레이션이 조기 종료되는 것을 막아주는 타이머 역할을 합니다.
  • ▸
    Test의 run_phase 시작 시 raise_objection을, 메인 Sequence 완료 후 drop_objection을 호출합니다. 카운트가 0이 되면 phase가 종료됩니다.

[9] 생성 & [10] Start

  • ▸
    Test 내부에서 검증 목적에 맞는 메인 Sequence 객체를 Factory를 통해 create() 합니다.
  • ▸
    생성된 Sequence를 어느 물리적 대상(Physical Sequencer)에서 실행할지 지정하여 start() 메서드를 호출, 트랜잭션 발생을 트리거합니다.

[11] Virtual Sequence

  • ▸
    단일 IP가 아닌 SoC 환경에서는 여러 Agent(예: AXI, APB)를 동시에 제어해야 합니다.
  • ▸
    자신만의 BFM/Driver 없이, 다른 Sub-Sequence들을 생성한 뒤 각각 알맞은 Physical Sequencer에 할당하여 실행 순서와 동기화를 지휘(Orchestrate)합니다.

4. 실행 옵션 및 결과 확인 (Runtime & Report)

UVM은 컴파일을 다시 하지 않고도 실행 인자(Runtime Option)만 변경하여 다양한 테스트를 수행할 수 있는 유연성을 제공합니다. 시뮬레이션이 종료되면 Report phase에서 Scoreboard와 Coverage 결과를 종합하여 최종 PASS/FAIL을 판정합니다.

[13] Plusarg / Runtime Option

  • ▸
    +UVM_TESTNAME=<name>: run_test() 함수와 연계되어, UVM Factory가 해당 문자열을 기반으로 Test 클래스를 찾아 인스턴스화하고 시뮬레이션을 시작하게 합니다.
  • ▸
    $value$plusargs: 터미널에서 +err_inj=1과 같이 입력한 커스텀 실행 변수를 런타임에 받아와, Test 내부의 조건문 분기나 시나리오 강도(Iteration 횟수 등)를 동적으로 조절하는 데 활용합니다.

[15] Report / Result Check

  • ▸
    데이터 수집: 시뮬레이션 종료 시점인 extract_phase 및 check_phase에서 Scoreboard의 Error 카운트나 Functional Coverage 달성도를 수집합니다.
  • ▸
    최종 판정: report_phase에서 `uvm_info, `uvm_error 매크로를 사용하여 결과를 로그(Log) 파일 및 콘솔에 명확하게 출력합니다. 에러가 0건일 때만 TEST PASSED를 선언합니다.

5. 전체 검증 환경 토폴로지 (TB Topology Overview)

Test 클래스는 전체 검증 환경(Testbench)을 담는 최상위 컨테이너입니다. 하위의 Env, Agent, Scoreboard 간의 계층 구조와 TLM(Transaction Level Modeling) 통신 채널을 시각적으로 구성합니다.

Active vs Passive Agent 구조

  • ▸
    UVM_ACTIVE: Test가 발생시킨 Sequence를 실제 DUT 신호로 변환하여 구동(Drive)해야 하는 경우, Sequencer와 Driver를 모두 활성화하여 생성합니다.
  • ▸
    UVM_PASSIVE: 단순히 DUT에서 출력되는 데이터를 관찰하고 수집만 하는 포트의 경우, 메모리 최적화를 위해 Monitor 컴포넌트만 단독으로 생성합니다.

TLM 연결 (Transaction Level Modeling)

  • ▸
    Test의 connect_phase에서 하위 컴포넌트 간의 통신 채널을 연결합니다.
  • ▸
    각 Agent의 Monitor가 버스의 신호를 Transaction 객체로 묶어 Analysis Port를 통해 Broadcast하면, Scoreboard가 이를 수신(Analysis Export)하여 기대값(Predictor)과 실제값을 비교(Compare)합니다.

6. UVM Phase Life-Cycle (Test 관점)

Test는 UVM의 표준 Phase 매커니즘을 관장합니다. 준비(Setup), 실행(Run), 정리 및 확인(Cleanup & Check)의 순차적인 흐름을 시각적으로 파악할 수 있습니다. run_phase만이 유일하게 시뮬레이션 시간을 소모(Time-consuming)합니다.

Setup Phase (함수, 시간 0)

시뮬레이션 시간이 흐르기 전(t=0)에 실행됩니다. 객체를 생성하는 build_phase는 부모에서 자식 방향(Top-Down)으로 실행되며, 포트를 연결하는 connect_phase는 반대로 자식에서 부모 방향(Bottom-Up)으로 실행되어 안정적인 조립을 보장합니다.

Run Phase (Task, 시간 진행)

유일하게 task로 정의되며 # delay나 @ (posedge clk)와 같이 실제 시뮬레이션 시간을 소모합니다. 모든 컴포넌트의 run_phase는 병렬(Fork-Join)로 동시 다발적으로 시작됩니다.

Cleanup Phase (함수, 시간 정지)

Objection이 모두 해제되어 시뮬레이션 메인 흐름이 종료된 후, 다시 t=End 시점에 멈춘 상태로 실행됩니다. Scoreboard에 남은 잔여 데이터를 추출(Extract)하고 데이터 결함 여부를 확인(Check)한 뒤, 최종 로그를 출력(Report)합니다.

7. Objection 매커니즘 심층 분석 (Objection & Drain Time)

UVM의 Phase(주로 run_phase) 종료를 제어하는 핵심 매커니즘입니다. 레퍼런스 카운팅(Reference Counting) 방식을 사용하며, 활성화된 Objection이 하나라도 남아있다면 해당 Phase는 계속 유지됩니다.

1. Raise & Drop 원리

여러 Sequence가 병렬로 실행될 때, 각각 시작 전 raise_objection을 호출하여 카운트를 1 증가시키고, 완료 후 drop_objection으로 1을 감소시킵니다. 카운트가 0보다 큰 동안 시뮬레이션은 종료되지 않습니다.

2. Drain Time (종료 유예)

Total Objection이 0이 된 직후 Phase를 바로 종료하면, DUT 내부 파이프라인에 남아있는 잔여 데이터(Flush)를 모니터가 수집하지 못할 수 있습니다. set_drain_time()을 통해 최종 Phase 종료를 일정 시간(예: 50ns) 유예할 수 있습니다.

3. 실무 최적화 가이드

하위 Sequence나 Driver 등 모든 컴포넌트 단위에서 Objection을 올리면 오버헤드가 발생해 시뮬레이션 속도가 저하됩니다. UVM 1.2+ 환경에서는 최상위 Test의 run_phase 또는 메인 Virtual Sequence 레벨에서만 단일하게 관리하는 것을 강력히 권장합니다.

8. 실제 코드 구현 예시 (SystemVerilog / UVM)

앞서 설명한 base_test 상속 구조, Factory Override, Config DB 설정, 그리고 Objection 매커니즘을 포함한 전체 코드 스니펫입니다. 

// -------------------------------------------------------------------------
// [Base Test] 모든 시나리오의 뼈대가 되는 공통 환경 설정
// -------------------------------------------------------------------------
class base_test extends uvm_test;
  `uvm_component_utils(base_test)

  uvm_env env;

  function new(string name = "base_test", uvm_component parent = null);
    super.new(name, parent);
  endfunction

  virtual function void build_phase(uvm_phase phase);
    super.build_phase(phase);
    // [3] Env 생성: UVM Factory 매커니즘 활용
    env = uvm_env::type_id::create("env", this);
    
    // [5] Config DB: 하위 컴포넌트로 Timeout 설정값 배포
    uvm_config_db#(int)::set(this, "env.*", "timeout_cnt", 10000);
  endfunction
endclass


// -------------------------------------------------------------------------
// [Scenario Test] 특정 시나리오를 구동하기 위한 파생 Test
// -------------------------------------------------------------------------
class test_scenario_A extends base_test;
  `uvm_component_utils(test_scenario_A)

  function new(string name = "test_scenario_A", uvm_component parent = null);
    super.new(name, parent);
  endfunction

  virtual function void build_phase(uvm_phase phase);
    // [6] Factory Override: 기존 정상 시퀀스(seq_A)를 Error 주입 시퀀스로 동적 교체
    seq_A::type_id::set_type_override(error_seq_A::get_type());
    super.build_phase(phase);
  endfunction

  // [7] Run Phase: 실제 시뮬레이션 시간이 흐르는 단계
  virtual task run_phase(uvm_phase phase);
    virtual_seq vseq;
    vseq = virtual_seq::type_id::create("vseq"); // [9] 시퀀스 생성
    
    // [Drain Time 설정] 전체 Objection이 0이 된 후 50ns 대기
    phase.phase_done.set_drain_time(this, 50ns);

    // [8] Objection 매커니즘 시작 (Reference Count +1)
    phase.raise_objection(this, "Scenario A Started");
    `uvm_info("TEST", "Executing Virtual Sequence...", UVM_LOW)
    
    // [10] 시퀀스 구동 트리거
    vseq.start(env.vsqr); 
    
    // [8] Objection 매커니즘 종료 (Reference Count -1)
    phase.drop_objection(this, "Scenario A Finished");
  endtask
endclass
