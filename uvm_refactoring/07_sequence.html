UVM Core Infrastructure

Sequencer : The Middle Route

Role: Router/Arbiter uvm_sequencer #(REQ, RSP)

1 [1, 5, 9] 핵심 역할과 Data Flow 파이프라인

Sequencer는 Sequence(자극 생성기)와 Driver(신호 주입기) 사이의 중개자(Router)입니다. 트랜잭션(REQ)을 전달하고, 응답(RSP)을 반환하는 양방향 데이터 파이프라인의 중심축을 담당합니다.

Expert Insight: TLM Pull & OOO Routing

Sequencer는 단순한 FIFO가 아닙니다. UVM TLM 1.0의 Pull 모델을 기반으로 하여, Driver가 처리 준비를 마치고 데이터를 당겨갈 때까지(get_next_item) 기다립니다. 또한 sequence_id와 transaction_id 속성을 추적하여, Out-of-Order (비순차적 파이프라인) 응답이 발생하더라도 정확한 부모 Sequence로 RSP를 라우팅하는 내부 sqr_rsp_analysis_fifo를 운용합니다.

2 [2, 3] 상속 구조 & REQ/RSP Type

Sequencer는 uvm_component의 계보를 이으며, 파라미터화된 클래스 구조를 가집니다. 통신할 데이터 타입(REQ, RSP)을 Parameter로 결정합니다.

m_sequencer vs p_sequencer:
Sequence가 실행될 때 내부적으로 m_sequencer(uvm_sequencer_base 타입) 핸들을 할당받습니다. 하지만 Register 모델이나 설정값에 접근하려면 User가 정의한 특정 Sequencer로의 다운캐스팅(Down-casting)이 필요하며, 이때 매크로를 통해 생성되는 것이 p_sequencer입니다.

3 [4, 6] seq_item_export & connect_phase

Agent의 connect_phase에서 Driver의 seq_item_port를 Sequencer의 seq_item_export와 연결하여 TLM 채널을 활성화합니다.

Pull Implementation:
이름은 'export'이지만, 실제 내부 클래스는 uvm_seq_item_pull_imp로 선언되어 있습니다. 일반적인 Push 통신(Monitor → Scoreboard)과 완전히 반대로 작동하며, Driver의 속도에 맞춰 시뮬레이션 타이밍을 동기화하는 가장 핵심적인 UVM 설계 패턴입니다.

4 [7, 8] Arbitration & Lock/Grab

다수의 시퀀스(예: 백그라운드 트래픽과 CPU 인터럽트)가 병렬로 구동될 때, Sequencer는 내부 Arbiter를 통해 충돌을 제어합니다.

Arbitration Policy

UVM은 UVM_SEQ_ARB_FIFO(기본값, 먼저 도착한 순서), WEIGHTED, RANDOM, STRICT_FIFO 등 다양한 중재 알고리즘을 지원합니다.

lock()

현재 큐의 맨 뒤에 대기하다가 차례가 오면 완료될 때까지 경로를 독점. (예: Read-Modify-Write 같은 원자적(Atomic) 초기화 작업 시 필수)

grab()

큐를 무시하고 최상단으로 새치기하여 독점. (예: 치명적인 에러 주입, 인터럽트 처리 등 긴급 상황). 남용 시 Deadlock 유발 위험.

5 [10, 11, 12] Virtual Sequencer & APIs

Virtual Sequencer Pattern

복잡한 SoC 검증에서는 물리적인 Driver와 연결되지 않는 Virtual Sequencer를 최상단에 둡니다. 이 컴포넌트는 하위 Agent들의 Sub-Sequencer 핸들들을 멤버로 가지고 있어, Virtual Sequence가 여러 IP의 동작을 오케스트레이션(Orchestration)할 수 있도록 허브 역할을 합니다.

[10] Debug & Control API

  • get_current_item() : 현재 처리중인 트랜잭션 검사 (Debug 용도)
  • is_blocked() : 큐가 lock/grab에 의해 정체되었는지 상태 반환
virtual_sequencer_pkg.sv 
// 1. Virtual Sequencer (Orchestrator)
class my_vsqr extends uvm_sequencer;
  `uvm_component_utils(my_vsqr)

  // 하위 물리적 Sequencer들에 대한 포인터 보유
  ahb_sequencer ahb_sqr;
  apb_sequencer apb_sqr;

  function new(string name, uvm_component parent);
    super.new(name, parent);
  endfunction
endclass

// 2. Virtual Sequence using p_sequencer
class my_vseq extends uvm_sequence;
  // m_sequencer를 my_vsqr 타입으로 다운캐스팅
  `uvm_declare_p_sequencer(my_vsqr)

  task body();
    // p_sequencer를 통해 특정 IP 시퀀서에 접근하여 실행
    ahb_sequence ahb_seq;
    ahb_seq = ahb_sequence::type_id::create("ahb_seq");
    
    // 큐 상태 확인 API 활용 (is_blocked)
    if (p_sequencer.ahb_sqr.is_blocked()) begin
      `uvm_warning("VSEQ", "AHB Sequencer is currently locked!")
    end
    
    ahb_seq.start(p_sequencer.ahb_sqr); 
  endtask
endclass
