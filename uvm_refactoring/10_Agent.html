UVM Architecture Guide

Agent : Integrated Verification Component

에이전트(Agent)는 DUT와의 통신을 전담하는 핵심 컨테이너입니다. Sequencer, Driver, Monitor를 캡슐화하여 모듈화된 검증 환경을 구성하며, Active/Passive 모드 전환을 통해 재사용성을 극대화합니다.

💡 UVM Expert's Note: 초보 검증 엔지니어는 종종 "왜 Env에 바로 Driver와 Monitor를 선언하지 않고 굳이 Agent로 한 번 더 감싸는가?"라는 의문을 갖습니다. Agent는 단순한 묶음이 아닌 '재사용의 최소 단위'입니다. IP 레벨 검증 환경을 칩 레벨(SoC)로 통합할 때, Agent 인터페이스 규격만 맞추면 기존 검증 컴포넌트를 단 한 줄의 코드 수정 없이 그대로 가져다 쓸 수 있도록 보장하는 가장 중요한 설계 패턴입니다.

1. 시스템 레벨 구조: Env ↔ Agent ↔ DUT

Agent의 역할 / Env와 Agent 연결 구조 / Virtual Interface 전달 / Config Object / Factory 구조

실무 설계 철학: 왜 uvm_config_db와 Factory를 사용하는가?

uvm_config_db를 통한 Virtual Interface(vif) 전달은 UVM의 핵심입니다. 하드웨어의 세계(RTL 핀)와 소프트웨어의 세계(Class)를 연결하는 유일한 다리 역할을 하며, Agent 내부의 Driver와 Monitor는 이 다리를 통해 DUT 핀 레벨의 신호를 트랜잭션(Transaction) 레벨로 추상화합니다.

또한 Factory 패턴을 사용해 내부 컴포넌트를 동적 생성함으로써, 향후 오류 주입(Error Injection) 테스트나 특별한 시나리오 검증 시 기존 원본 코드를 일절 건드리지 않고 Agent나 Driver 클래스만 상속받은 커스텀 클래스로 손쉽게 덮어쓰기(Override) 할 수 있습니다.

2. Agent 내부 구성 및 동작 모드 (Active vs Passive)

내부 구성 / uvm_active_passive_enum / Sequencer ↔ Driver 연결 / Monitor AP / Agent AP

실무 설계 철학: Active / Passive 전환의 진짜 의미

Active/Passive 모드 분리는 '검증 환경 재사용성의 꽃'입니다. 블록 레벨(Block-level) 환경에서는 DUT에 직접 패킷을 밀어넣기 위해 Active 모드(SQR, DRV 활성화)로 동작시킵니다.

하지만 SoC 시스템 레벨 환경으로 통합되면, 실제 CPU나 다른 마스터 IP가 패킷을 인가합니다. 이때 기존 검증 환경의 Agent를 Passive 모드로만 전환해주면, 시뮬레이션 중에 Driver가 버스를 점유하여 발생하는 충돌을 방지하면서도 기존에 작성한 Monitor, Checker, Coverage 모델은 100% 그대로 살려 재사용할 수 있습니다.

3. 동작 모드 결정: get_is_active() 메커니즘

Config DB Override / uvm_active_passive_enum 결정 트리 / Base Class Automation

실무 설계 철학: 왜 new()를 쓰지 않고 Factory를 쓰는가?

UVM에서 객체를 메모리에 올릴 때 SystemVerilog의 내장 함수인 new()를 직접 사용하지 않고 type_id::create()를 호출하는 것은 강력한 규약입니다. 이는 동적 타입 교체(Type Override)를 위함입니다. 만약 new()로 하드코딩해버리면, 나중에 오류를 주입하는 my_error_driver로 교체하고 싶을 때 Agent 원본 코드를 직접 수정해야 합니다. 반면 Factory 엔진을 사용하면 원본 코드를 건드리지 않고 외부(Test)에서 한 줄의 코드로 컴포넌트를 바꿔치기 할 수 있습니다.

또한 create("name", this)에서 두 번째 인자로 전달되는 this 포인터는 매우 중요합니다. 이는 생성되는 자식 컴포넌트(SQR, DRV)에게 "너의 부모는 나(Agent)다"라고 알려주는 역할을 하며, 이 정보들이 모여 우측의 UVM 위상 트리(Topology Tree)를 완성하게 됩니다. 이 트리가 있어야만 향후 uvm_info 출력 시 경로(예: uvm_test_top.env.agent.drv)가 정확히 찍히고 Top-Down 페이즈 제어가 가능해집니다.

5. Phase 실행 흐름 메커니즘

get_is_active() / build_phase() / connect_phase()

실무 설계 철학: Phase 별 구조 제어 메커니즘

UVM Phase 메커니즘에서 build_phase는 Top-Down(위에서 아래로)으로 실행되고, connect_phase는 Bottom-Up(아래에서 위로)으로 실행됩니다. 즉, Agent의 build_phase 단계에서 메모리에 할당할 객체를 확정 지어야만 자식 컴포넌트들의 phase가 정상적으로 이어서 실행될 수 있습니다. 따라서 get_is_active() 체크를 build_phase에서 미리 수행하여 불필요한 인스턴스(SQR, DRV) 생성을 원천 차단하는 것이 메모리 최적화와 에러 방지의 핵심입니다.

6. 실전 코드 구조 및 상속

uvm_agent 상속 구조 / Factory 등록 / Phase 구현체 코드

my_agent.sv 
class my_agent extends uvm_agent;
    // Factory Registration
    `uvm_component_utils(my_agent)

    // Component Handles (내부 구성 요소 선언)
    my_sequencer  sqr;
    my_driver     drv;
    my_monitor    mon;
    
    // Agent-level Analysis Port 선언
    uvm_analysis_port #(my_transaction) ap;

    // Constructor
    function new(string name, uvm_component parent);
        super.new(name, parent);
    endfunction

    // Build Phase (컴포넌트 생성 및 설정)
    virtual function void build_phase(uvm_phase phase);
        super.build_phase(phase);
        
        // Monitor는 Active/Passive 관계없이 항상 생성
        mon = my_monitor::type_id::create("mon", this);
        ap  = new("ap", this);

        // Active 모드일 경우에만 SQR, DRV 생성
        if (get_is_active() == UVM_ACTIVE) begin
            sqr = my_sequencer::type_id::create("sqr", this);
            drv = my_driver::type_id::create("drv", this);
        end
    endfunction

    // Connect Phase (TLM 및 인터페이스 연결)
    virtual function void connect_phase(uvm_phase phase);
        super.connect_phase(phase);
        
        // Monitor AP를 Agent AP에 연결 (상위 Env로 전달하기 위함)
        mon.ap.connect(ap);

        // Active 모드일 경우 Sequencer와 Driver 연결
        if (get_is_active() == UVM_ACTIVE) begin
            drv.seq_item_port.connect(sqr.seq_item_export);
        end
    endfunction

endclass

실무 설계 철학: 캡슐화(Encapsulation)와 포트 연결

위 코드를 보면 agent.ap라는 자체 Analysis Port를 생성하고, 내부에 있는 mon.ap와 연결(connect)하는 것을 볼 수 있습니다. Env나 Scoreboard가 agent.mon.ap와 같이 Agent 내부 구현 깊숙한 곳에 직접 접근하게 코딩하는 것은 결합도(Coupling)를 높이는 안 좋은 습관(Anti-pattern)입니다. Agent는 자신의 포트인 것처럼 데이터를 감싸서 외부로 노출시켜야 내부 Monitor 구조가 변경되더라도 상위 Env가 영향을 받지 않습니다.

7. 데이터 브로드캐스팅: Analysis Port 메커니즘

Publisher-Subscriber 패턴 / 1:N 통신 / write() 함수 호출 / 컴포넌트 Decoupling

왜 Analysis Port인가? (1:N 통신)

일반적인 TLM Port(예: get_port, put_port)는 1:1 통신만 가능하며, 연결된 목적지가 없으면 시뮬레이션 중 Fatal Error를 발생시킵니다. 반면 uvm_analysis_port는 0명부터 N명까지 다수의 구독자(Subscriber)에게 동일한 데이터를 뿌려줄 수 있는 브로드캐스팅 전용 포트입니다.

실무 설계 철학: 완벽한 Decoupling

Monitor는 ap.write(tr)를 호출해 패킷을 밖으로 던질 뿐, 밖에서 Scoreboard가 그것을 받아 비교를 하는지, Coverage 모델이 커버리지를 채우는지 전혀 알 필요가 없습니다. 심지어 아무도 연결되어 있지 않아도 에러 없이 정상 동작합니다. 이 강력한 분리(Decoupling) 특성 덕분에 검증 컴포넌트들을 블록처럼 자유롭게 조립할 수 있습니다.
