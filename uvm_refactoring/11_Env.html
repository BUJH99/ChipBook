UVM Architecture Guide

Env : Integrate Verification Components

uvm_env는 검증 환경의 최상위 컨테이너입니다. Agent, Scoreboard, Reference Model 등 하위 컴포넌트들을 인스턴스화하고(Build), 이들 간의 통신 채널을 연결(Connect)하는 중추적인 역할을 시각적으로 파악해보세요.

[1, 3, 6, 7, 9] UVM Environment Core Architecture

Env의 역할과 내부 구성요소 간의 유기적인 데이터 흐름 (TLM Connections)

[3] Agent의 구성

DUT 인터페이스 프로토콜 처리를 전담합니다. Active Agent는 Stimulus 구동(Driver/Seq)과 모니터링을 모두 수행하며, Passive Agent는 모니터링만 수행하여 리소스를 최적화합니다.

[7] Reference Model

입력 트랜잭션을 기반으로 DUT가 출력해야 할 정확한 기대값(Expected Data)을 생성하는 골든 모델입니다. 주로 C/C++나 SystemVerilog로 구현된 Predictor 역할을 수행합니다.

[6] Scoreboard / Coverage

Scoreboard는 Ref Model의 기대값과 실제 DUT 출력값을 비교(Check)합니다. Coverage Collector는 검증 목표 달성도를 정량적으로 측정하기 위해 커버리지를 샘플링합니다.

[9] Virtual Sequencer

여러 개의 하위 Sequencer들을 중앙에서 제어하여 복잡한 시스템 레벨 시나리오를 연출합니다. 직접 아이템을 생성하지 않고 다른 Sequencer로 트래픽을 분배합니다.

[2] uvm_env 상속구조

uvm_env는 uvm_component를 상속받습니다. 이는 Env가 검증 환경 계층 구조에서 고정된 물리적 위치를 가지며, 시뮬레이션 Phase 주기에 따라 순차적으로 동작함을 의미합니다.

[8] Env Config (uvm_config_db)

Dynamic Hierarchy: UVM 환경은 고정되어 있지 않습니다. 최상위 Test에서 검증 시나리오에 맞는 env_cfg(설정 객체)를 조립하여 글로벌 데이터베이스(uvm_config_db)에 set 합니다.

이후 Env는 자신의 build_phase에서 이 설정값을 get 하여 읽어 들인 뒤, 조건문(if)에 따라 내부의 Agent나 Scoreboard를 메모리에 동적으로 할당할지 말지 결정합니다. 이를 통해 하나의 Env 코드로 다양한 검증 모드(Top-level, Block-level, Active/Passive)를 완벽히 커버할 수 있습니다.

[4, 5, 10] Phase별 Env 역할 메커니즘

[4] build_phase()

Top-Down 실행: Config 수신 & UVM Factory 메모리 할당

Top-Down 원칙: 상위 컴포넌트(Env)가 먼저 생성되고, 이후 하위 컴포넌트(Agent, SB)가 생성됩니다. 상위 계층에서 uvm_config_db::set()으로 내려보낸 설정(Virtual Interface, 동작 모드 등)을 하위 객체가 안전하게 get()하여 자신의 구조를 결정할 수 있게 합니다.

Factory 패턴: UVM에서는 일반적인 new() 생성자 대신 type_id::create() 메서드를 사용합니다. 이를 통해 검증 환경 코드를 수정하지 않고도 외부 테스트 시나리오에서 내부 객체를 다른 타입으로 덮어쓰기(Override)할 수 있는 엄청난 유연성을 확보합니다.

[5] connect_phase()

Bottom-Up 실행: TLM 포트/소켓 결합 및 포인터 매핑

Bottom-Up 원칙: 하위 인스턴스들의 build_phase가 모두 완료되어 메모리(객체)가 확실히 생성된 이후에 실행됩니다. 이는 아직 할당되지 않은 Null 포인터 객체에 접근하여 발생하는 에러를 구조적으로 원천 차단합니다.

TLM Channel Binding: 서로 다른 하위 컴포넌트 간의 통신을 위해 connect() 메서드를 사용합니다. 이 과정은 송신 측의 Port(플러그)에 수신 측 Export/Imp(소켓)의 메모리 주소(Handle)를 등록하여 고속 트랜잭션 패스를 완성하는 작업입니다.

end_of_elaboration_phase()

Bottom-Up 실행: 계층 구조 확정(Lock) 및 최종 점검

Hierarchy Lock: 이 Phase에 진입하면 UVM 프레임워크는 현재까지 생성된 모든 컴포넌트의 계층 구조를 잠금(Lock) 처리합니다. 이후에는 새로운 컴포넌트를 동적으로 인스턴스화하거나 구조를 변경할 수 없게 되어, 시뮬레이션 안정성이 보장됩니다.

Topology Printing: 모든 구조 생성과 포트 연결이 정상적으로 완료되었는지 시뮬레이션 시간 0(Zero-time)에서 최종 확인하는 단계입니다. 테스트 벤치 최상위에서 uvm_top.print_topology()를 호출하여 전체 컴포넌트 트리와 포트 결합 상태를 터미널 창에 시각적으로 출력합니다.

run_phase(uvm_phase phase)

Parallel 실행: 병렬 스레드 구동 및 Objection 제어

Time-Consuming Task: build, connect 등 0초(Zero-time)에 실행되는 함수(Function) Phase와 달리, 유일하게 시간(Time)을 소모하며 실제 검증 시나리오를 구동하는 Task Phase입니다.

Parallel & Objection: Env 하위의 모든 컴포넌트(Seq, Drv, Mon, SB)들의 run_phase가 fork~join 형태로 동시에 병렬(Parallel) 실행됩니다. 또한 raise_objection과 drop_objection을 통해 모든 스레드의 작업이 안전하게 완료될 때까지 시뮬레이션 종료를 지연시킵니다.

[11] Hierarchical Env (Vertical Reuse)

블록 레벨(Block-level) Env를 시스템 레벨(System-level) Env에서 재사용하는 계층적 통합 아키텍처

수직적 재사용 (Vertical Reuse)의 핵심

UVM Env는 그 자체로 독립적인 검증 환경(Block-level)이 될 수 있으며, 코드 수정 없이 상위 시스템 환경(System-level Env)의 하위 컴포넌트로 그대로 재사용(Instance)될 수 있습니다. 위 다이어그램처럼 이미 개발 검증이 완료된 MAC Env와 PHY Env를 System Env 내부에 블록 쌓듯 통합함으로써 검증 환경 개발 리소스를 극적으로 절약할 수 있습니다.

System Virtual Sequencer 할당 로직

최상위의 System Virtual Sequencer는 자신이 직접 트랜잭션을 생성하지 않고, 내부에 선언된 빈 포인터 변수(Handle)에 하위 블록(MAC, PHY)의 실제 물리적 Sequencer 주소를 할당(Assign) 받습니다. 이를 통해 SoC 레벨의 단일 통합 시나리오(Virtual Sequence)에서 MAC과 PHY에 동시에 복합적인 트래픽을 인가하는 오케스트레이션(Orchestration)이 가능해집니다.

System-Level Scoreboard 연동

블록 내부에 Local Scoreboard가 존재하더라도, 시스템 레벨의 데이터 정합성(End-to-End Checking)을 확인하기 위해 하위 Env의 Monitor Port를 최상위 System Scoreboard에 직접 연결(Connect)할 수 있습니다. 이는 계층 구조를 뚫고 데이터를 유연하게 라우팅할 수 있는 TLM(Transaction Level Modeling)의 강력함을 보여줍니다.

[12] Register Model (RAL) Integration in Env

Env 내부에서 추상화된 레지스터 모델(RAL), Adapter, Predictor가 물리적 Bus Agent와 연동되는 양방향 아키텍처

uvm_reg_block (RAL) & Map

디자인의 실제 하드웨어 레지스터 맵을 검증 환경 내부에 추상화한 가상 메모리 공간입니다. Env의 connect_phase에서 default_map.set_sequencer()를 호출하여 하위의 물리적 버스 Agent(Sequencer 및 Adapter)와 바인딩됩니다. 시퀀스에서 rm.CTRL.write()를 호출하면 이 맵을 타고 데이터가 내려갑니다.

uvm_reg_adapter (변환기)

UVM 프레임워크가 생성하는 범용 레지스터 객체(uvm_reg_item)와 사용자가 정의한 특정 버스 프로토콜 트랜잭션(예: APB/AXI bus_item) 간의 형 변환을 담당합니다. Top-Down 시에는 reg2bus()로 버스 시퀀스 아이템을 생성하고, 예측을 위해서는 bus2reg()를 제공하는 필수 브릿지입니다.

uvm_reg_predictor & Mirror

Bus Agent의 Monitor가 캡처한 '실제 버스 트래픽'을 Analysis Port를 통해 수신합니다. 이를 Adapter의 bus2reg()를 이용해 다시 레지스터 트랜잭션으로 해석한 뒤, RAL 모델 내부의 Mirror Value를 실제 하드웨어 상태와 완벽하게 동기화하여 추후 정합성 검사(Checking)에 사용합니다.

[13] UVM Env Code Template

위에서 설명한 UVM Env의 계층 구조와 Phase 동작 원리가 실제 SystemVerilog(SV) 코드에서 구현되는 템플릿입니다.

// my_env.sv
class my_env extends uvm_env;

  // 1. 하위 컴포넌트 및 Config 핸들 선언
  my_agent       agent;
  my_scoreboard  sb;
  my_env_config  cfg;

  // 2. UVM Component Automation Macro 등록
  `uvm_component_utils(my_env)

  // 3. 생성자 (Constructor)
  function new(string name = "my_env", uvm_component parent = null);
    super.new(name, parent);
  endfunction

  // ---------------------------------------------------------
  // [4] Build Phase (Top-Down Execution)
  // ---------------------------------------------------------
  virtual function void build_phase(uvm_phase phase);
    super.build_phase(phase);
    
    // [8] Config DB에서 설정 객체(cfg) 가져오기 (get)
    if (!uvm_config_db#(my_env_config)::get(this, "", "env_cfg", cfg)) begin
      `uvm_fatal("ENV/NOCFG", "No configuration specified for my_env")
    end

    // 설정값(cfg)에 따라 서브 컴포넌트를 Factory를 통해 동적 생성 (create)
    if (cfg.has_agent) begin
      agent = my_agent::type_id::create("agent", this);
    end
    
    if (cfg.has_sb) begin
      sb = my_scoreboard::type_id::create("sb", this);
    end
  endfunction

  // ---------------------------------------------------------
  // [5] Connect Phase (Bottom-Up Execution)
  // ---------------------------------------------------------
  virtual function void connect_phase(uvm_phase phase);
    super.connect_phase(phase);
    
    // Agent의 Analysis Port(ap)를 Scoreboard의 Analysis Export(exp)와 물리적 연결
    if (cfg.has_agent && cfg.has_sb) begin
      agent.ap.connect(sb.exp);
    end
  endfunction

endclass : my_env

Summary Checkpoints

  • ✓ Env는 UVM Verification Topology의 컨테이너입니다. (Not a stimulus generator)
  • ✓ build_phase에서 모든 서브 컴포넌트를 생성합니다.
  • ✓ connect_phase에서 TLM 포트 연결을 수행하여 데이터 패스를 완성합니다.
  • ✓ Active Agent는 Stimulus를, Passive Agent는 Monitoring만 수행합니다.
  • ✓ Virtual Sequencer는 환경 내 여러 Agent의 Sequencer들을 제어합니다.
